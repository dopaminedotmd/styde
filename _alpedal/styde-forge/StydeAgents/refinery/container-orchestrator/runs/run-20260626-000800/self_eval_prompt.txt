## 1. Multi-Stage Docker Builds

### 1.1 Node.js — Build & Run

```dockerfile
# =============================================================================
# NODE.JS MULTI-STAGE DOCKERFILE
# =============================================================================
# Stage 1: Dependencies & Build
# -----------------------------------------------------------------------------
FROM node:20-alpine AS builder

# Install build toolchain for native addons (bcrypt, node-sass, etc.)
RUN apk add --no-cache python3 make g++

WORKDIR /app

# Copy package manifests first — cache layer for npm ci
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies for build)
RUN npm ci

# Copy source and compile
COPY tsconfig.json ./
COPY src/ ./src/
RUN npm run build

# Prune devDependencies after build, keeping only production deps
RUN npm prune --production

# =============================================================================
# Stage 2: Production Runtime
# -----------------------------------------------------------------------------
FROM node:20-alpine AS runtime

# Create non-root user
RUN addgroup -g 1001 -S appgroup && \
    adduser  -u 1001 -S appuser -G appgroup

WORKDIR /app

# Copy only production artifacts from builder
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /app/dist       ./dist
COPY --from=builder --chown=appuser:appgroup /app/package.json ./

# Expose app port
EXPOSE 3000

# Switch to non-root user
USER appuser

# Health check
HEALTHCHECK --interval=15s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

ENTRYPOINT ["node", "dist/index.js"]
```

### 1.2 Python — Build & Run

```dockerfile
# =============================================================================
# PYTHON MULTI-STAGE DOCKERFILE
# =============================================================================
# Stage 1: Build virtualenv and install deps
# -----------------------------------------------------------------------------
FROM python:3.12-slim AS builder

# Install build deps (postgres driver, lxml, etc.)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    libxml2-dev \
    libxslt1-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Create virtualenv in a predictable location
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Install dependencies into the venv
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application source
COPY src/ ./src/

# =============================================================================
# Stage 2: Slim runtime
# -----------------------------------------------------------------------------
FROM python:3.12-slim AS runtime

# Install only runtime libs (no build toolchain)
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
    && rm -rf /var/lib/apt/lists/*

# Create non-root user
RUN groupadd -r appgroup && useradd -r -g appgroup appuser

# Copy virtualenv from builder (the entire site-packages tree)
COPY --from=builder --chown=appuser:appgroup /opt/venv /opt/venv
COPY --from=builder --chown=appuser:appgroup /app/src /app/src

# Set venv as default python path
ENV PATH="/opt/venv/bin:$PATH"

WORKDIR /app

EXPOSE 8000

USER appuser

HEALTHCHECK --interval=15s --timeout=5s --start-period=10s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

ENTRYPOINT ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 3.6 Health Probes

```yaml
# =============================================================================
# HEALTH PROBES — Reference
# =============================================================================
# Three probe types, each configurable:

# ── Startup Probe ──────────────────────────────────────────────────────────
#   • Runs FIRST (before liveness/readiness). Protects slow-starting apps.
#   • Once it succeeds, it never runs again for the pod's lifetime.
#   • If it fails, the container is killed and restarted.
#   • Ideal for: DB migrations, cache warm-up, model loading.
startupProbe:
  httpGet:
    path: /health/startup
    port: 8000
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 12        # 5s × 12 = 60s max startup time

# ── Liveness Probe ─────────────────────────────────────────────────────────
#   • Runs repeatedly after startup succeeds.
#   • If it fails → container is killed and restarted.
#   • Ideal for: detecting deadlocks, hung processes.
#   • Warning: set conservative thresholds; too aggressive = restart loops.
livenessProbe:
  httpGet:
    path: /health/live
    port: 8000
  periodSeconds: 15
  timeoutSeconds: 5
  failureThreshold: 3          # 3 consecutive failures → restart

# ── Readiness Probe ────────────────────────────────────────────────────────
#   • Runs repeatedly after startup succeeds.
#   • If it fails → pod is removed from Service endpoints (no traffic).
#   • Pod is NOT restarted on readiness failure.
#   • Ideal for: DB connection lost, downstream dependency unhealthy.
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
  periodSeconds: 10
  timeoutSeconds: 3
  failureThreshold: 3
  successThreshold: 1          # 1 success after failure restores traffic

# ── Alternative probe types ────────────────────────────────────────────────
# TCP socket probe:
#   tcpSocket:
#     port: 8000
#
# Exec probe:
#   exec:
#     command: ["/bin/sh", "-c", "pg_isready -h localhost -p 5432"]
#
# gRPC probe (K8s 1.24+):
#   grpc:
#     port: 50051
#     service: health.v1.Health
```

**Example application health endpoints** (FastAPI):

```python
# health.py — add to your FastAPI app
from fastapi import APIRouter
from app.db import check_database_connection

router = APIRouter(tags=["health"])


@router.get("/health/startup")
async def startup_health():
    """Called by the startup probe. Signal ready ONLY after all init is done.
    Return non-200 until migrations, cache warm-up, etc. are complete."""
    return {"status": "ok"}


@router.get("/health/live")
async def liveness_health():
    """Called by the liveness probe. The simplest possible check.
    Just verify the process isn't hung — no DB, no external calls."""
    return {"status": "alive"}


@router.get("/health/ready")
async def readiness_health():
    """Called by the readiness probe. Check downstream dependencies.
    If the DB is down, return 503 to stop receiving traffic."""
    db_ok = await check_database_connection()
    if not db_ok:
        return JSONResponse(
            status_code=503,
            content={"status": "not ready", "reason": "database unreachable"}
        )
    return {"status": "ready"}
```

### 3.7 Resource Limits & Requests

```yaml
# =============================================================================
# RESOURCE MANAGEMENT — Embedded in container spec
# =============================================================================
resources:
  # ── Requests ─────────────────────────────────────────────────────────────
  #   • What the scheduler RESERVES on a node for this container.
  #   • Used for bin-packing: a pod only fits on a node with free request
  #     capacity.
  #   • Set requests close to steady-state usage.
  requests:
    cpu: 250m        # 0.25 CPU cores (guaranteed minimum)
    memory: 256Mi    # 256 MiB RAM (guaranteed minimum)

  # ── Limits ───────────────────────────────────────────────────────────────
  #   • Hard ceiling. CPU: throttled if exceeded. Memory: OOMKilled.
  #   • Set limits above burst/peak usage but below node capacity.
  limits:
    cpu: 500m        # 0.5 CPU cores max (burstable)
    memory: 512Mi    # 512 MiB max (OOMKilled if exceeded)

# ── QoS Classes ────────────────────────────────────────────────────────────
#   • Guaranteed:  requests == limits for both CPU and memory → highest priority
#   • Burstable:   requests < limits, or requests set but limits missing
#   • BestEffort:  no requests or limits set → first to be evicted
```

**Right-sizing guide**:

| Workload Type       | Initial Requests      | Initial Limits       | Notes                              |
|---------------------|-----------------------|----------------------|------------------------------------|
| Node.js API         | cpu: 200m, mem: 256Mi | cpu: 500m, mem: 512Mi | Adjust after 3 days of metrics    |
| Python FastAPI      | cpu: 250m, mem: 256Mi | cpu: 1, mem: 512Mi    | Python needs more CPU headroom    |
| Background Worker   | cpu: 100m, mem: 128Mi | cpu: 500m, mem: 512Mi | Burstable CPU for queue spikes    |
| PostgreSQL          | cpu: 1, mem: 2Gi     | cpu: 2, mem: 4Gi      | Use StatefulSet, not Deployment   |
| Redis               | cpu: 100m, mem: 256Mi | cpu: 500m, mem: 512Mi | Monitor maxmemory                 |

```yaml
spec:
  # ── Target ──────────────────────────────────────────────────────────────
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend-api

  # ── Scale Bounds ────────────────────────────────────────────────────────
  minReplicas: 3               # never scale below this
  maxReplicas: 20              # never scale above this

  # ── Behavior (scaling policies) ─────────────────────────────────────────
  behavior:
    # Scale-up behavior
    scaleUp:
      stabilizationWindowSeconds: 60        # wait 60s before scaling up
      policies:
        - type: Percent
          value: 100                         # double the replicas
          periodSeconds: 60
        - type: Pods
          value: 4                           # or add 4 pods at most
          periodSeconds: 60
      selectPolicy: Max                      # use the policy that adds MORE pods

    # Scale-down behavior
    scaleDown:
      stabilizationWindowSeconds: 300        # wait 5 min before scaling down
      policies:
        - type: Percent
          value: 50                          # remove at most 50% of pods
          periodSeconds: 60
        - type: Pods
          value: 2                           # or remove at most 2 pods
          periodSeconds: 60
      selectPolicy: Min                      # use the policy that removes FEWER pods

  # ── Metrics ─────────────────────────────────────────────────────────────
  metrics:
    # CPU-based scaling
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70          # target 70% CPU utilization

    # Memory-based scaling
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80          # target 80% memory utilization

    # Custom metric: requests per second (requires Prometheus Adapter)
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "500"             # 500 req/s per pod

    # External metric: queue depth (requires Prometheus Adapter)
    - type: External
      external:
        metric:
          name: queue_depth
          selector:
            matchLabels:
              queue: email-jobs
        target:
          type: AverageValue
          averageValue: "100"             # scale on queue depth > 100
```

## Best Practices Summary

| Category            | Practice                                                              |
|---------------------|-----------------------------------------------------------------------|
| **Images**          | Use multi-stage builds; pin base image SHAs; run as non-root          |
| **Secrets**         | Never check into Git; use External Secrets Operator or Sealed Secrets |
| **Probes**          | Always configure all three probes; keep liveness lightweight          |
| **Resources**       | Always set requests & limits; use VPA in recommendation mode first   |
| **HPA**             | Combine CPU + memory + custom metrics; set conservative scale-down    |
| **Security**        | `readOnlyRootFilesystem: true`; drop all capabilities; use PSP/PSA   |
| **Networking**      | Use NetworkPolicies for east-west traffic; TLS everywhere             |
| **Observability**   | Structured JSON logging to stdout; Prometheus /metrics endpoint       |
| **Graceful Shutdown** | Catch SIGTERM; use `terminationGracePeriodSeconds`; preStop hooks  |
| **PodDisruptionBudget** | Set PDB alongside HPA to protect voluntary disruptions            |
| **Topology**        | Use `topologySpreadConstraints` for HA across zones                  |
