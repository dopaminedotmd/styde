## 1. Cycle 2 Overview — Why These Five Patterns

| Pattern | Problem It Solves | Risk Without It |
|---------|-------------------|-----------------|
| **Pod Disruption Budget** | Voluntary disruptions (node drains, cluster autoscaler) can evict all replicas simultaneously | Brief outage while pods reschedule; SLA breach during maintenance windows |
| **Topology Spread Constraints** | Pods colocated on same node/zone despite anti-affinity gaps | Single-zone failure takes down all replicas |
| **RuntimeClass (gVisor/Kata)** | Container escape risks in multi-tenant or untrusted-workload clusters | Kernel-shared exploit chains; CVE-2024-21626-style host compromise |
| **Vertical Pod Autoscaler** | Manual resource sizing is guesswork; OOMKills or massive waste | 40–60% CPU waste in typical clusters; unpredictable OOMs |
| **Gateway API** | Ingress is L7-only, annotation-hell, no cross-namespace delegation | Brittle routing; no role separation between cluster-admin and app-dev |

Each pattern below includes: **the problem**, **the YAML manifest**, **the verification steps**, and **common failure modes**.

## 2. Pod Disruption Budgets

### 2.1 Concept

A **PodDisruptionBudget (PDB)** caps the number of pods that can be **voluntarily** disrupted simultaneously (node drain, cluster autoscaler scale-in, `kubectl drain`). It does **not** protect against involuntary disruptions (node crash, OOMKill, liveness failure).

PDB uses one of two selectors:

- **`maxUnavailable`** — "at most N pods may be down"
- **`minAvailable`** — "at least N pods must stay up"

You choose one. For stateful workloads, prefer `maxUnavailable: 1`. For stateless, prefer `minAvailable` as a percentage.

### 2.2 Manifest

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: payment-service-pdb
  namespace: production
spec:
  # minAvailable: "80%"       # percentage-based — safest for variable replicas
  maxUnavailable: 1            # absolute — safest for stateful workloads
  selector:
    matchLabels:
      app: payment-service
  # unhealthyPodEvictionPolicy: IfHealthyBudget  # K8s 1.27+ (default: IfHealthyBudget)
```

### 2.3 Best Practices

| Scenario | Recommendation |
|----------|---------------|
| Stateless deployment, 3+ replicas | `minAvailable: "75%"` — tolerates one drain cycle per quorum |
| StatefulSet with PVCs (e.g., Kafka, PostgreSQL) | `maxUnavailable: 1` — never drain two at once, attach/detach races |
| Single-replica workloads | PDB is useless — drain always blocks; use priority classes instead |
| Cluster autoscaler with spot instances | `minAvailable: "50%"` + `unhealthyPodEvictionPolicy: AlwaysAllow` (K8s 1.27+) |
| Zero-downtime deployments with rolling update | Set `maxSurge` > 0 on Deployment so PDB + surge don't deadlock |

### 2.4 Verification

```bash
# Check PDB status
kubectl get pdb -n production payment-service-pdb

# Expected output:
# NAME                     MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
# payment-service-pdb      N/A             1                 1                     5m

# Simulate drain (dry-run)
kubectl drain node-01 --dry-run=server --delete-emptydir-data

# If PDB blocks, output shows:
# error: cannot evict pod as it would violate the pod's disruption budget.
```

### 2.5 Common Failure Modes

1. **Deadlock with rolling updates:** Deployment `maxSurge=0` + PDB `maxUnavailable=0` → no new node can schedule new pod before old one terminates.
   - **Fix:** Set `maxSurge=1` on Deployment.
2. **PDB blocks cluster-autoscaler scale-in** indefinitely because pods are stuck on nodes marked for drain.
   - **Fix:** Add `unhealthyPodEvictionPolicy: AlwaysAllow` (K8s 1.27+) for pods that are CrashLoopBackOff.
3. **Wrong selector:** PDB matches pods in a different namespace — silently useless.
   - **Fix:** Always `kubectl get pods -l <selector> --all-namespaces` before applying PDB.

## 3. Topology Spread Constraints

### 3.1 Concept

**Topology Spread Constraints** distribute pods evenly across failure domains (zones, nodes, racks) using a `maxSkew` tolerance. Unlike `podAntiAffinity` (hard or soft binary rule), topology spread is a **best-effort distribution** with a configurable skew.

Key fields:

- **`topologyKey`** — the failure domain label (e.g., `topology.kubernetes.io/zone`, `kubernetes.io/hostname`)
- **`maxSkew`** — the maximum allowed imbalance between the min and max domain
- **`whenUnsatisfiable`** — `DoNotSchedule` (hard) or `ScheduleAnyway` (soft)
- **`minDomains`** — minimum number of domains that must exist before constraint activates (K8s 1.25+)

### 3.2 Manifest

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-gateway
  namespace: production
spec:
  replicas: 6
  selector:
    matchLabels:
      app: api-gateway
  template:
    metadata:
      labels:
        app: api-gateway
    spec:
      topologySpreadConstraints:
        # Spread across zones (coarse-grained)
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: api-gateway
          minDomains: 2                   # K8s 1.25+: skip constraint if <2 zones exist

        # Spread across nodes (fine-grained)
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway  # soft — schedule anyway if impossible
          labelSelector:
            matchLabels:
              app: api-gateway

      containers:
        - name: api-gateway
          image: registry.example.com/api-gateway:v2.4.0
          ports:
            - containerPort: 8080
```

### 3.3 Layout Outcome (3 zones × 3 nodes)

```
Zone us-east-1a (nodes: a1, a2, a3)
  api-gateway-xxxxx → node a1
  api-gateway-xxxxx → node a2

Zone us-east-1b (nodes: b1, b2, b3)
  api-gateway-xxxxx → node b1
  api-gateway-xxxxx → node b2

Zone us-east-1c (nodes: c1, c2, c3)
  api-gateway-xxxxx → node c1
  api-gateway-xxxxx → node c2
```

Skew = 0. All 6 replicas evenly distributed.

### 3.4 When to Use vs. podAntiAffinity

| Requirement | Use |
|-------------|-----|
| Every replica must be on a different node | `podAntiAffinity: requiredDuringScheduling` |
| Spread evenly, but colocation is OK when unavoidable | `topologySpreadConstraints` with `ScheduleAnyway` |
| Strict zone spread with no skew tolerance | `topologySpreadConstraints` with `DoNotSchedule` + `maxSkew=0` |
| DaemonSet-style (one per node) | DaemonSet — don't use either |

### 3.5 Verification

```bash
# Check distribution across zones
kubectl get pods -n production -l app=api-gateway -o wide \
  | awk '{print $7}' | sort | uniq -c

# Expected:
#   2 us-east-1a
#   2 us-east-1b
#   2 us-east-1c

# Check scheduling events for skew violations
kubectl get events -n production --field-selector reason=FailedScheduling \
  | grep "topology spread"
```

### 3.6 Common Failure Modes

1. **`minDomains` mismatch:** Constraint requires 3 zones but cluster has only 2 → pods Pending indefinitely. Use `minDomains` to set a floor or `ScheduleAnyway` as safety valve.
2. **Node autoscaler races:** New nodes take 2–4 minutes to join; scheduler sees `DoNotSchedule` skew violation and refuses pending pods. Use `ScheduleAnyway` on hostname spread.
3. **Label inconsistency:** Nodes missing `topology.kubernetes.io/zone` label → constraint ignored. Verify with `kubectl get nodes --show-labels`.

## 4. RuntimeClass — gVisor & Kata Containers

### 4.1 Concept

**RuntimeClass** selects an alternative container runtime at the Pod level. Instead of the default `runc` (shared kernel with host), you can use:

| Runtime | Isolation Model | Overhead | Use Case |
|---------|----------------|----------|----------|
| **runc** (default) | Shared kernel, namespace isolation | 0% | Trusted workloads |
| **gVisor** | User-space kernel (sentry), syscall interception | 5–15% perf | Semi-trusted CI/CD, web-facing |
| **Kata Containers** | Lightweight VM per pod (KVM), hardware isolation | 10–30% perf | Multi-tenant SaaS, untrusted code exec |
| **Firecracker** | microVM, KVM-based, faster than Kata | 5–10% bootstrap | FaaS, sandboxed user functions |

### 4.2 Prerequisites

```bash
# Install gVisor on nodes
# https://gvisor.dev/docs/user_guide/install/

# Verify containerd config
cat /etc/containerd/config.toml | grep -A5 "gvisor\|kata"

# Create RuntimeClass objects in cluster
kubectl apply -f runtime-classes.yaml
```

### 4.3 RuntimeClass Definitions

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc                    # CRI handler name in containerd config
scheduling:
  nodeSelector:                   # restrict to nodes with gVisor installed
    runtime: gvisor
  tolerations:
    - key: "runtime"
      operator: "Equal"
      value: "gvisor"
      effect: "NoSchedule"
---
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata
handler: kata
scheduling:
  nodeSelector:
    runtime: kata
  tolerations:
    - key: "runtime"
      operator: "Equal"
      value: "kata"
      effect: "NoSchedule"
---
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata-confidential    # Kata 3.x+ with TDX/SEV confidential computing
handler: kata-tdx
scheduling:
  nodeSelector:
    runtime: kata-tdx
  tolerations:
    - key: "runtime"
      operator: "Equal"
      value: "kata-tdx"
      effect: "NoSchedule"
```

### 4.4 Pod Using gVisor

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sandboxed-microservice
  namespace: production
spec:
  runtimeClassName: gvisor
  containers:
    - name: app
      image: registry.example.com/sandboxed-app:v1.3.0
      ports:
        - containerPort: 3000
      resources:
        requests:
          cpu: "500m"
          memory: "256Mi"
        limits:
          cpu: "1000m"
          memory: "512Mi"
```

### 4.5 Pod Using Kata Containers (Untrusted Code Execution)

```yaml

  name: code-executor
  namespace: sandbox

  runtimeClassName: kata             # hardware-level isolation
  shareProcessNamespace: false       # kata VM boundary prevents this anyway
  containers:

      securityContext:
        allowPrivilegeEscalation: false

            - ALL

          memory: "1Gi"
```

### 4.6 RuntimeClass in Deployments / Admission Controllers

```yaml
# Set default RuntimeClass cluster-wide via PodDefaults (K8s 1.28+) or Kyverno
# Kyverno ClusterPolicy — mutate all pods in 'sandbox' namespace to use gVisor

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: enforce-sandbox-namespace-gvisor
spec:
  rules:
    - name: set-runtime-class
      match:
        any:
          - resources:
              kinds:
                - Pod
              namespaces:
                - sandbox
                - untrusted
      mutate:
        patchStrategicMerge:
          spec:
            runtimeClassName: gvisor
```

### 4.7 Verification

```bash
# Confirm runtime class on a running pod
kubectl get pod sandboxed-microservice -n production \
  -o jsonpath='{.spec.runtimeClassName}'
# Expected: gvisor

# Verify actual runtime via node
kubectl exec -n production sandboxed-microservice -- dmesg 2>&1 | head -5
# gVisor: "Starting gVisor..."
# Kata:   QEMU/KVM guest kernel messages

# Node-level verification
crictl pods --label io.kubernetes.runtimeclass=gvisor
```

### 4.8 Common Failure Modes

1. **Node missing runtime:** Pod stuck in `ContainerCreating` with event `failed to create containerd task: runtime "runsc" not found`.
   - **Fix:** Add nodeSelector to RuntimeClass `scheduling` block to prevent scheduling onto nodes without the runtime.
2. **Syscall incompatibility:** `mount()`, `ptrace()`, or `perf_event_open()` blocked by gVisor sentry. Application crashes with `EPERM`.
   - **Fix:** gVisor allowslist specific syscalls via `--allow-syscall` flag in runsc config. Review gVisor compatibility guide before onboarding.
3. **Performance regression:** Kata boots a full VM per pod (~300–800ms). Not suitable for bursty <100ms-latency workloads.
   - **Fix:** Use gVisor for moderate isolation with better perf; use Kata only for untrusted/Multi-tenancy workloads.

## 5. Vertical Pod Autoscaler

### 5.1 Concept

The **Vertical Pod Autoscaler (VPA)** automatically adjusts CPU and memory **requests/limits** based on historical usage. Unlike HPA (horizontal: add/remove replicas), VPA resizes existing pods.

Three components:

| Component | Role |
|-----------|------|
| **Recommender** | Watches metrics history, produces recommended requests/limits |
| **Updater** | Evicts pods if they need resizing (only in `Auto` mode) |
| **Admission Controller** | Mutates pod resource spec at creation time (only in `Auto` mode) |

Four update modes:

| Mode | Behavior | Use Case |
|------|----------|----------|
| `Off` | Only produce recommendations, no action | Audit / planning |
| `Initial` | Set resources at pod creation, no eviction | New workloads you want right-sized from start |
| `Auto` | Evict pod to apply new values (respects PDB!) | Production, with PDB safety net |
| `Recreate` | Evict-only, Admission Controller doesn't mutate | Legacy migration path |

### 5.2 Installation

```bash
# Via Helm
helm repo add fairwinds-stable https://charts.fairwinds.com/stable
helm upgrade --install vpa fairwinds-stable/vpa \
  --namespace kube-system \
  --set recommender.enabled=true \
  --set updater.enabled=true \
  --set admissionController.enabled=true

# Verify
kubectl get pods -n kube-system | grep vpa
# vpa-admission-controller-xxx   1/1 Running
# vpa-recommender-xxx            1/1 Running
# vpa-updater-xxx                1/1 Running
```

### 5.3 VPA Manifest

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: payment-processor-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: payment-processor

  updatePolicy:
    updateMode: "Auto"
    minReplicas: 2              # K8s 1.29+: only evict if ≥2 replicas running

  resourcePolicy:
    containerPolicies:
      - containerName: app
        controlledResources: ["cpu", "memory"]
        minAllowed:
          cpu: "250m"
          memory: "256Mi"
        maxAllowed:
          cpu: "4000m"
          memory: "8Gi"
        controlledValues: RequestsAndLimits  # K8s 1.30: control both
      - containerName: sidecar-proxy
        mode: "Off"             # Exclude sidecar from VPA management
```

### 5.4 Checking Recommendations

```bash

kubectl describe vpa payment-processor-vpa -n production

# Recommendation:
#   Container Recommendations:
#     Container Name:  app
#     Lower Bound:
```

### 5.5 VPA + HPA Coexistence

VPA and HPA can coexist but must be configured carefully:

```yaml
---
# VPA: manage CPU/memory requests
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
      - containerName: app
        controlledResources: ["memory"]     # VPA manages memory only
        minAllowed:
          memory: "256Mi"
        maxAllowed:
          memory: "4Gi"
---
# HPA: manage replica count based on CPU
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70            # VPA adjusts requests → HPA target stays proportional
```

**Rule:** If HPA uses CPU as a metric, VPA must NOT manage CPU — the feedback loops interfere. Either:
- VPA manages memory, HPA manages CPU + replicas (recommended)
- VPA manages both but HPA uses custom metrics (e.g., Prometheus request latency)

### 5.6 Common Failure Modes

1. **Pod churn:** VPA in `Auto` mode evicts pods repeatedly because recommended values bounce. Mitigate with `minReplicas: 2` and wider resource bounds.
2. **OOMKill loop:** VPA recommends `500Mi` but app spikes to `800Mi`. Set `maxAllowed.memory` high enough for known spikes.
3. **Incompatible with StatefulSets:** VPA eviction breaks StatefulSet identity ordering. Use `Off` mode for monitoring, apply manually via GitOps.
4. **Conflict with custom metrics:** VPA updater sees recommendation drift but actual usage is controlled by application-level limits (e.g., JVM `-Xmx`). Set container `minAllowed` = current value to suppress false drift.

## 6. Gateway API — Replacing Ingress

### 6.1 Concept

**Gateway API** is the successor to Ingress, graduating to GA in K8s 1.30. It separates concerns across roles:

| Role | Gateway API Resource | Responsibility |
|------|---------------------|----------------|
| **Infrastructure Provider** | `GatewayClass` | Defines controller implementation (Istio, Envoy, NGINX Gateway Fabric, Traefik) |
| **Cluster Operator** | `Gateway` | Deploys load balancer, configures TLS certificates, sets allowed routes |
| **Application Developer** | `HTTPRoute`, `GRPCRoute`, `TCPRoute`, `TLSRoute` | Defines routing rules, header matching, traffic splitting, retries |

### 6.2 Comparison: Ingress vs Gateway API

| Feature | Ingress | Gateway API |
|---------|---------|-------------|
| **L7 routing** | path-based host-based | ✓ (plus header, query, method matching) |
| **L4 routing** | ✗ (or via ConfigMap hacks) | TCPRoute, UDPRoute (GA 1.30) |
| **gRPC routing** | ✗ | GRPCRoute (GA 1.30) |
| **Traffic splitting** | Annotation per controller | Native `backendRefs[].weight` |
| **Header-based routing** | Annotation per controller | Native `matches[].headers` |
| **Cross-namespace routing** | ✗ | `ReferenceGrant` |
| **Role separation** | ✗ (one resource) | GatewayClass → Gateway → HTTPRoute |
| **TLS SNI passthrough** | ✗ | TLSRoute with `passthrough` mode |

### 6.3 Infrastructure Setup

#### GatewayClass (installed by infra team or Helm chart)

```yaml

```

```yaml
            - name: X-Client-Type

      backendRefs:
        - name: payment-service-mobile
          port: 8080
          weight: 1

    # Default backend
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: payment-service-stable

        - type: RequestHeaderModifier

            value: /api

        backendRequest: 8s
      retry:
        attempts: 3
        backoff: 100ms
        codes: [500, 502, 503]
```

### 7.3 Consolidated Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
  namespace: payment
  annotations:
    vpa-autoscaling/enabled: "true"           # signals VPA admission controller
spec:
  replicas: 6
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2        # Allow surge to avoid PDB deadlock
      maxUnavailable: 1  # Conservative: never have <5 pods
  selector:
    matchLabels:
      app: payment-service
  template:
    metadata:
      labels:
        app: payment-service
    spec:
      runtimeClassName: gvisor            # 🔒 sandbox via gVisor

      topologySpreadConstraints:          # 📍 spread across zones + nodes
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: payment-service
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app: payment-service

      containers:
        - name: app
          image: registry.example.com/payment-service:v3.1.2
          ports:
            - containerPort: 8080
              name: http
            - containerPort: 9090
              name: metrics
          env:
            - name: K8S_NODE_NAME
              valueFrom:
                fieldRef:
                  fieldPath: spec.nodeName
          resources:
            requests:                        # VPA-managed baseline
              cpu: "500m"
              memory: "512Mi"
            limits:
              cpu: "2000m"
              memory: "2Gi"
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /readyz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
```

### 7.4 Apply Order

```bash
# 1. RuntimeClass first (pods depend on it)
kubectl apply -f runtime-class.yaml

# 2. Namespace + RBAC
kubectl apply -f namespace.yaml

# 3. PDB (before Deployment to catch new pods)
kubectl apply -f pdb.yaml

# 4. VPA (starts observing immediately)
kubectl apply -f vpa.yaml

# 5. Deployment + Service
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# 6. HPA (needs Deployment to exist)
kubectl apply -f hpa.yaml

# 7. Gateway API routes
kubectl apply -f gateway-route.yaml
kubectl apply -f reference-grant.yaml

# 8. Verify everything
kubectl get deploy,pdb,vpa,hpa,httproute -n payment
```

## 8. Operational Runbooks

### 8.1 Node Maintenance (Drain with PDB)

```bash
#!/bin/bash
# drain-node.sh — safely drain a node respecting PDBs

NODE=$1
echo "=== Pre-drain checks for $NODE ==="

# List all PDBs on this node
kubectl get pods --all-namespaces -o wide --field-selector spec.nodeName=$NODE \
  --no-headers | while read ns pod rest; do
  kubectl get pdb -n $ns -o name 2>/dev/null
done | sort -u

# Dry-run drain
kubectl drain $NODE --dry-run=server --ignore-daemonsets --delete-emptydir-data 2>&1

echo "=== If blocked by PDB, stagger evictions ==="

# Cordon first (no new pods)
kubectl cordon $NODE

# Drain with a 5-minute deadline
kubectl drain $NODE \
  --ignore-daemonsets \
  --delete-emptydir-data \
  --grace-period=300 \
  --timeout=10m
```

### 8.2 Diagnosing VPA Eviction Storms

```bash
# Check VPA eviction history
kubectl get events -n payment --field-selector reason=EvictedByVPA \
  --sort-by='.lastTimestamp'

# Check if PDB is blocking VPA evictions
kubectl describe vpa payment-service-vpa -n payment | grep -A5 "Conditions"

# Sample blocked condition:
# Conditions:
#   Last Transition Time:  2026-06-26T00:15:00Z
#   Status:                True
#   Type:                  ConfigUnsupported
#   Message:               Cannot evict pods: PDB payment-service-pdb blocks eviction

# If this happens, temporarily relax PDB:
kubectl patch pdb payment-service-pdb -n payment \
  -p '{"spec":{"maxUnavailable":3}}' --type=merge
# Wait for VPA evictions to complete, then restore
```

### 8.3 Gateway API Debugging

```bash
# Check if Gateway has an address
kubectl get gateway production-gateway -n gateway-infra -o yaml | grep -A2 addresses

# Check HTTPRoute acceptance
kubectl get httproute payment-api-routes -n payment -o yaml | grep -A10 "status:"

# Sample problems:
# - "Accepted: False" → check parentRef Gateway exists and sectionName matches

```

## Appendix: Comparison Matrix — c1 vs c2 Delta

| Dimension | c1 (Foundation) | c2 (Advanced) |
|-----------|----------------|---------------|
| **Workload Resilience** | ReplicaSet, basic liveness/readiness | PDB blocks drain storms, topology spread survives zone failures |
| **Security Isolation** | Namespace + NetworkPolicy | RuntimeClass — gVisor (user-kernel) or Kata (VM-level) |
| **Resource Tuning** | Manual requests/limits | VPA — automatic sizing with bounds, PDB-safe evictions |
| **Ingress/Routing** | Ingress (nginx annotation hell) | Gateway API — HTTPRoute, GRPCRoute, traffic splitting, header matching, role separation |
| **Availability** | HPA for horizontal | HPA + VPA coexistence pattern |
| **Maintenance Windows** | Blind drains | PDB-aware drains, `unhealthyPodEvictionPolicy` |
| **Multi-tenancy** | Namespaces only | RuntimeClass isolation + Gateway API cross-namespace ReferenceGrant |

### Complexity Budget

| Pattern | Lines of YAML | Control Plane Overhead | Risk of Misconfiguration |
|---------|:------------:|:----------------------:|:------------------------:|
| PDB | ~15 | None | Low |
| TopologySpreadConstraints | ~20 | Scheduler scoring | Medium (minDomains) |
| RuntimeClass | ~50 | Per-pod overhead (5–30%) | Medium (node labels) |
| VPA | ~30 | Recommender + Updater pods | High (eviction storms, HPA conflict) |
| Gateway API | ~200 | New CRDs + controller | Medium (ReferenceGrant bypass) |
