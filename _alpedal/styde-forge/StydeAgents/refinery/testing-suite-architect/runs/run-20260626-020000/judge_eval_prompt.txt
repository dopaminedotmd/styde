## 1. Playwright Component Testing

### 1.1 Architecture & Trade-offs

Playwright Component Testing (CT) mounts your components in a **real browser** (Chromium, Firefox, WebKit) instead of JSDOM or happy-dom. This eliminates the "it works in tests but not in the browser" class of bugs. However, it is slower than jsdom-based Testing Library tests.

**When to use Playwright CT vs. Testing Library (jsdom):**

| Concern | Testing Library + jsdom | Playwright CT |
|---|---|---|
| Speed | ~10–50ms per test | ~200–800ms per test |
| Browser API fidelity | Limited (must mock) | Full (real browser) |
| CSS layout / rendering | No real layout | Real rendering |
| Visual screenshots | Not possible | Built-in |
| Web API support (IndexedDB, WebSocket, etc.) | Must mock | Native |
| CI complexity | Simple | Requires browser binaries |
| Debugging | Node debugger | Playwright Inspector + Trace Viewer |

**Recommendation**: Use Testing Library for fast unit/integration tests (80% of tests). Use Playwright CT for components that depend on browser APIs (drag-and-drop, canvas, media, CSS animations, complex forms).

### 1.2 Mounting React Components

#### Installation

```typescript
  // Retry flaky tests on CI

    ['html', { outputFolder: 'playwright-ct-report' }],
    ['json', { outputFile: 'playwright-ct-report/results.json' }],
```

```typescript
// ============================================================
// src/components/LoginForm.ct.spec.tsx
// Playwright Component Test — LoginForm in a real browser
// ============================================================
import { test, expect } from '@playwright/experimental-ct-react';
import { LoginForm } from './LoginForm';

test.describe('LoginForm', () => {
  // ── Rendering ───────────────────────────────────────────────
  test('renders email and password fields', async ({ mount }) => {
    const component = await mount(<LoginForm />);

    await expect(component.getByLabel('Email')).toBeVisible();
    await expect(component.getByLabel('Password')).toBeVisible();
    await expect(component.getByRole('button', { name: 'Sign In' })).toBeVisible();
  });

  // ── Interaction ─────────────────────────────────────────────
  test('shows validation errors on empty submit', async ({ mount }) => {
    const component = await mount(<LoginForm />);

    // Click submit without filling fields
    await component.getByRole('button', { name: 'Sign In' }).click();

    // Browser-native validation or custom error messages
    await expect(component.getByText('Email is required')).toBeVisible();
    await expect(component.getByText('Password is required')).toBeVisible();
  });

  // ── Callback Props ──────────────────────────────────────────
  test('calls onSubmit with form values', async ({ mount }) => {
    let submittedData: { email: string; password: string } | null = null;

    const component = await mount(
      <LoginForm
        onSubmit={(data) => {
          submittedData = data;
        }}
      />
    );

    await component.getByLabel('Email').fill('user@example.com');
    await component.getByLabel('Password').fill('s3cret!');
    await component.getByRole('button', { name: 'Sign In' }).click();

    // Playwright CT runs in the same process — we can assert on closures
    expect(submittedData).toEqual({
      email: 'user@example.com',
      password: 's3cret!',
    });
  });

  // ── Loading State ───────────────────────────────────────────
  test('disables button during submission', async ({ mount }) => {
    // Component accepts an `isSubmitting` prop for testing
    const component = await mount(<LoginForm isSubmitting={true} />);

    const button = component.getByRole('button', { name: 'Signing in…' });
    await expect(button).toBeDisabled();
  });
});
```

### 1.3 Interaction Testing in Real Browsers

Playwright CT provides the full Playwright interaction API, which is more powerful and reliable than `@testing-library/user-event` for complex interactions.

```typescript
// ============================================================
// src/components/DragAndDropList.ct.spec.tsx
// Real drag-and-drop — impossible in jsdom
// ============================================================
import { test, expect } from '@playwright/experimental-ct-react';
import { DragAndDropList } from './DragAndDropList';

const ITEMS = [
  { id: '1', text: 'Buy groceries' },
  { id: '2', text: 'Walk the dog' },
  { id: '3', text: 'Read a book' },
];

test('reorders items via drag and drop', async ({ mount, page }) => {
  const component = await mount(<DragAndDropList items={ITEMS} />);

  const firstItem = component.locator('[data-item-id="1"]');
  const thirdItem = component.locator('[data-item-id="3"]');

  // Real browser drag-and-drop
  await firstItem.dragTo(thirdItem);

  // Verify new order
  const itemTexts = component.locator('[data-testid="item-text"]');
  await expect(itemTexts).toHaveText([
    'Walk the dog',
    'Read a book',
    'Buy groceries',
  ]);
});

test('supports keyboard reordering', async ({ mount }) => {
  const component = await mount(<DragAndDropList items={ITEMS} />);

  // Focus first item
  await component.locator('[data-item-id="1"]').focus();

  // Move down with keyboard
  await component.locator('[data-item-id="1"]').press('Space'); // Grab
  await component.locator('[data-item-id="1"]').press('ArrowDown'); // Move
  await component.locator('[data-item-id="1"]').press('ArrowDown'); // Move again
  await component.locator('[data-item-id="1"]').press('Space'); // Drop

  const itemTexts = component.locator('[data-testid="item-text"]');
  await expect(itemTexts).toHaveText([
    'Walk the dog',
    'Read a book',
    'Buy groceries',
  ]);
});
```

### 1.4 Network Mocking at the Component Level

Playwright CT runs in a real browser — you can intercept network requests using Playwright's `page.route()` API, which is far more realistic than MSW when you need to test actual fetch behavior, response timing, or error handling at the network layer.

```typescript
// ============================================================
// src/components/UserProfile.ct.spec.tsx
// Real network interception in component tests
// ============================================================
import { test, expect } from '@playwright/experimental-ct-react';
import { UserProfile } from './UserProfile';

test.describe('UserProfile — with network mocking', () => {
  test('displays user data from API', async ({ mount, page }) => {
    // Intercept the fetch call the component makes
    await page.route('**/api/users/me', async (route) => {
      await route.fulfill({
        status: 200,
        contentType: 'application/json',
        body: JSON.stringify({
          id: 'usr_1',
          name: 'Ada Lovelace',
          email: 'ada@example.com',
          avatarUrl: 'https://example.com/avatars/ada.jpg',
        }),
      });
    });

    const component = await mount(<UserProfile />);

    // Wait for the data to load
    await expect(component.getByText('Ada Lovelace')).toBeVisible();
    await expect(component.getByText('ada@example.com')).toBeVisible();

    // Avatar image loaded
    const avatar = component.getByRole('img', { name: 'Ada Lovelace' });
    await expect(avatar).toHaveAttribute('src', 'https://example.com/avatars/ada.jpg');
  });

  test('shows error state on network failure', async ({ mount, page }) => {
    await page.route('**/api/users/me', async (route) => {
      await route.abort('failed');
    });

    const component = await mount(<UserProfile />);

    await expect(component.getByText(/failed to load/i)).toBeVisible();
    await expect(component.getByRole('button', { name: /retry/i })).toBeVisible();
  });

  test('handles slow responses gracefully', async ({ mount, page }) => {
    await page.route('**/api/users/me', async (route) => {
      // Simulate 2-second delay before response
      await new Promise((resolve) => setTimeout(resolve, 2000));
      await route.fulfill({
        status: 200,
        body: JSON.stringify({ id: '1', name: 'Slow User' }),
      });
    });

    const component = await mount(<UserProfile />);

    // Skeleton loader during fetch
    await expect(component.locator('.skeleton-loader')).toBeVisible();

    // Eventually resolves
    await expect(component.getByText('Slow User')).toBeVisible({ timeout: 5000 });
  });

  test('handles malformed JSON response', async ({ mount, page }) => {
    await page.route('**/api/users/me', async (route) => {
      await route.fulfill({
        status: 200,
        contentType: 'application/json',
        body: '{invalid json!!!!',
      });
    });

    const component = await mount(<UserProfile />);

    await expect(component.getByText(/unexpected error/i)).toBeVisible();
  });
});
```

### 1.6 Accessibility Checks

```typescript

// src/components/Modal.ct.spec.tsx
// Accessibility checks in Playwright CT
// ============================================================
import { test, expect } from '@playwright/experimental-ct-react';
import AxeBuilder from '@axe-core/playwright';
import { Modal } from './Modal';

test.describe('Modal — accessibility', () => {
  test('passes axe-core audit when open', async ({ mount, page }) => {
    await mount(
      <Modal isOpen={true} title="Confirm Delete" onClose={() => {}}>
        <p>Are you sure you want to delete this item?</p>
        <button>Cancel</button>
        <button>Delete</button>
      </Modal>
    );

    const results = await new AxeBuilder({ page })
      .include('[role="dialog"]')
      .analyze();

    expect(results.violations).toEqual([]);
  });

  test('traps focus within the modal', async ({ mount, page }) => {
    const component = await mount(
      <Modal isOpen={true} title="Settings" onClose={() => {}}>
        <button>Tab 1</button>
        <button>Tab 2</button>
        <button>Tab 3</button>
      </Modal>
    );

    // Focus should start on the first focusable element
    await expect(component.getByRole('button', { name: 'Tab 1' })).toBeFocused();

    // Tab through all buttons — should cycle within modal
    await page.keyboard.press('Tab');
    await expect(component.getByRole('button', { name: 'Tab 2' })).toBeFocused();

    await page.keyboard.press('Tab');
    await expect(component.getByRole('button', { name: 'Tab 3' })).toBeFocused();

    // Should wrap back to first
    await page.keyboard.press('Tab');
    await expect(component.getByRole('button', { name: 'Tab 1' })).toBeFocused();
  });

  test('closes on Escape key', async ({ mount }) => {
    let closed = false;
    const component = await mount(
      <Modal
        isOpen={true}
        title="Dismissible"
        onClose={() => {
          closed = true;
        }}
      >
        <p>Press Escape to close</p>
      </Modal>
    );

    await component.press('Escape');
    expect(closed).toBe(true);
  });
});
```

## 2. Visual Regression with Percy & Chromatic

### 2.1 Percy Setup & Configuration

[Percy](https://percy.io/) by BrowserStack captures DOM snapshots and performs pixel-by-pixel visual diffs. It works with Playwright, Cypress, Storybook, and any framework.

#### Installation

```bash
npm install --save-dev @percy/cli @percy/playwright
```

#### Percy Configuration (`.percy.yml`)

```yaml
# ============================================================
# .percy.yml — Percy visual testing configuration
# ============================================================
version: 2

snapshot:
  # Widths to capture (responsive testing)
  widths:
    - 375    # Mobile
    - 768    # Tablet
    - 1280   # Desktop
    - 1440   # Wide desktop

  # Minimum height of each snapshot
  min-height: 1024

  # Enable JavaScript before capturing (for SPAs)
  enable-javascript: true

  # Percy CSS — hide dynamic elements that cause false diffs
  percy-css: |
    /* Hide timestamps */
    .timestamp, [data-testid="relative-time"] { visibility: hidden !important; }
    /* Hide animated spinners */
    .spinner, .loading-indicator { display: none !important; }
    /* Hide randomized placeholder images */
    img[src*="placeholder"] { visibility: hidden !important; }

  # Discovery options for SPAs
  discovery:
    # Wait for network to be idle before capturing
    network-idle-timeout: 250
    # Additional selectors to wait for before capturing
    wait-for-selectors:
      - '[data-percy-ready]'
      - '.content-loaded'
    # Allowed hostnames for asset discovery
    allowed-hostnames:
      - 'cdn.example.com'
      - 'images.example.com'

# Static snapshot configuration (for Storybook, style guides)
static-snapshots:
  base-url: /storybook
  snapshot-files: '**/storybook-static/**/*.html'
  ignore-files: '**/storybook-static/iframe.html'
```

### 2.4 Handling Dynamic Content & Anti-Flake Patterns

Visual regression is notorious for false positives. These patterns eliminate common sources of flake.

```typescript
// ============================================================
// e2e/visual/anti-flake-helpers.ts
// Reusable patterns for stable visual testing
// ============================================================
import type { Page } from '@playwright/test';

/**
 * Stabilize the page before taking a visual snapshot.
 * Call this before any toHaveScreenshot() or percySnapshot().
 */
export async function stabilizePage(page: Page): Promise<void> {
  // ── 1. Wait for network idle ───────────────────────────────
  await page.waitForLoadState('networkidle');

  // ── 2. Wait for fonts to load ─────────────────────────────
  await page.evaluate(() => document.fonts.ready);

  // ── 3. Hide all animations and transitions ─────────────────
  await page.addStyleTag({
    content: `
      *, *::before, *::after {
        animation-duration: 0s !important;
        animation-delay: 0s !important;
        transition-duration: 0s !important;
        transition-delay: 0s !important;
      }
    `,
  });

  // ── 4. Replace dynamic dates with fixed values ─────────────
  await page.evaluate(() => {
    const now = new Date('2026-06-26T12:00:00Z');
    // Override Date.now for deterministic timestamps
    const OriginalDate = Date;
    // Note: Date.now override doesn't affect `new Date()` — use
    // a fixed seed or mock at the application level for true determinism
  });

  // ── 5. Hide scrollbars (OS-specific styling) ──────────────
  await page.addStyleTag({
    content: `
      ::-webkit-scrollbar { display: none; }
      * { scrollbar-width: none; }
    `,
  });

  // ── 6. Set consistent system fonts ────────────────────────
  await page.addStyleTag({
    content: `
      body {
        font-family: 'Inter', -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif !important;
      }
    `,
  });

  // ── 7. Blur any active elements (remove focus rings) ───────
  await page.evaluate(() => {
    (document.activeElement as HTMLElement)?.blur();
  });

  // ── 8. Small delay for layout to settle ────────────────────
  await page.waitForTimeout(300);
}

/**
 * Replace dynamic content with stable placeholders before snapshot.
 */
export async function sanitizeDynamicContent(page: Page): Promise<void> {
  await page.evaluate(() => {
    // Replace relative timestamps
    document.querySelectorAll('[data-testid="relative-time"]').forEach((el) => {
      el.textContent = '2 hours ago';
    });

    // Replace randomized avatar URLs with stable ones
    document.querySelectorAll('.user-avatar img').forEach((el, i) => {
      (el as HTMLImageElement).src = `https://placehold.co/40x40/${i % 2 ? '00f' : '0f0'}/fff?text=U`;
    });

    // Remove "NEW" badges that appear/disappear based on date
    document.querySelectorAll('.badge-new').forEach((el) => {
      el.classList.remove('badge-new');
      el.classList.add('badge-new--visible');
    });

    // Freeze CSS countdown timers
    document.querySelectorAll('[data-countdown]').forEach((el) => {
      el.setAttribute('data-countdown-frozen', el.textContent || '00:00');
    });
  });
}
```

## 3. Contract Testing with Pact

### 3.1 Consumer-Driven Contract Fundamentals

[Pact](https://pact.io/) is a consumer-driven contract testing framework. The **consumer** defines expectations about an API, Pact records a contract (pact file), and the **provider** verifies it can fulfill those expectations.

**Why contract testing instead of (or in addition to) E2E tests:**

| Concern | E2E Tests | Contract Tests (Pact) |
|---|---|---|
| Requires both services running | Yes | No — mock on consumer side |
| Speed | Slow (seconds to minutes) | Fast (milliseconds) |
| Catches API breakage | Only if E2E covers the scenario | Yes — provider verification catches it |
| Deployment coupling | Must wait for both services | Independent deployability |
| Maintenance | Brittle, end-to-end | Each side maintains its own |

#### Installation

```bash
npm install --save-dev @pact-foundation/pact @pact-foundation/pact-node
```

### 3.2 Writing Consumer Tests

```typescript
// ============================================================
// consumer/tests/user-service.pact.spec.ts
// Consumer contract: The frontend app expects certain API behavior
// ============================================================
import path from 'node:path';
import {
  PactV4,
  MatchersV3,
  SpecificationVersion,
  type V4UnconfiguredInteraction,
  type V4ConsumerPact,
} from '@pact-foundation/pact';
import { like, eachLike, string, integer, boolean } from '@pact-foundation/pact/dsl/matchers';

import { UserService } from '../services/user-service';

// ── Pact Setup ────────────────────────────────────────────────
const provider: V4ConsumerPact = PactV4.create({
  consumer: 'web-app',
  provider: 'user-service',
  spec: SpecificationVersion.SPECIFICATION_VERSION_V4,
  dir: path.resolve(__dirname, '../pacts'),
  logLevel: process.env.CI ? 'error' : 'info',
});

describe('UserService (Consumer)', () => {
  describe('GET /users/:id', () => {
    it('fetches a user by ID', async () => {
      // ── Arrange: Define expected interaction ──────────────
      await provider
        .given('a user with id "usr_42" exists')
        .uponReceiving('a request for user details')
        .withRequest('GET', '/users/usr_42', (builder) => {
          builder.headers({ Accept: 'application/json' });
        })
        .willRespondWith(200, (builder) => {
          builder
            .headers({ 'Content-Type': 'application/json' })
            .jsonBody({
              id: like('usr_42'),
              name: like('Ada Lovelace'),
              email: like('ada@example.com'),
              avatarUrl: like('https://cdn.example.com/avatars/ada.jpg'),
              role: string('developer'),
              isActive: boolean(true),
              createdAt: like('2024-01-15T10:30:00Z'),
              tags: eachLike('typescript', 2),
            });
        })
        .executeTest(async (mockserver) => {
          // ── Act: Call the real service against the mock ───
          const userService = new UserService(mockserver.url);
          const user = await userService.getUser('usr_42');

          // ── Assert: Verify the service handles the response correctly ──
          expect(user).toEqual({
            id: 'usr_42',
            name: 'Ada Lovelace',
            email: 'ada@example.com',
            avatarUrl: 'https://cdn.example.com/avatars/ada.jpg',
            role: 'developer',
            isActive: true,
            createdAt: '2024-01-15T10:30:00Z',
            tags: ['typescript', 'typescript'],
          });
        });
    });

    it('handles 404 when user not found', async () => {
      await provider
        .given('no user with id "usr_999" exists')
        .uponReceiving('a request for a non-existent user')
        .withRequest('GET', '/users/usr_999', (builder) => {
          builder.headers({ Accept: 'application/json' });
        })
        .willRespondWith(404, (builder) => {
          builder.jsonBody({
            error: like('User not found'),
            code: string('USER_NOT_FOUND'),
          });
        })
        .executeTest(async (mockserver) => {
          const userService = new UserService(mockserver.url);

          await expect(userService.getUser('usr_999')).rejects.toThrow('User not found');
        });
    });
  });

  describe('POST /users', () => {
    it('creates a new user', async () => {
      await provider
        .given('user creation is allowed')
        .uponReceiving('a request to create a new user')
        .withRequest('POST', '/users', (builder) => {
          builder
            .headers({
              'Content-Type': 'application/json',
              Accept: 'application/json',
            })
            .jsonBody({
              name: like('Charles Babbage'),
              email: like('charles@example.com'),
              role: string('analyst'),
            });
        })
        .willRespondWith(201, (builder) => {
          builder
            .headers({ Location: like('/users/usr_new') })
            .jsonBody({
              id: like('usr_new'),
              name: like('Charles Babbage'),
              email: like('charles@example.com'),
              role: string('analyst'),
              isActive: boolean(true),
              createdAt: like('2026-06-26T12:00:00Z'),
              tags: eachLike('new-user'),
            });
        })
        .executeTest(async (mockserver) => {
          const userService = new UserService(mockserver.url);
          const user = await userService.createUser({
            name: 'Charles Babbage',
            email: 'charles@example.com',
            role: 'analyst',
          });

          expect(user.id).toBeTruthy();
          expect(user.name).toBe('Charles Babbage');
          expect(user.role).toBe('analyst');
        });
    });

    it('handles validation errors', async () => {
      await provider
        .given('user creation validation is strict')
        .uponReceiving('a request with invalid email')
        .withRequest('POST', '/users', (builder) => {
          builder.jsonBody({
            name: like('No Email'),
            email: string('not-an-email'),
            role: string('developer'),
          });
        })
        .willRespondWith(422, (builder) => {
          builder.jsonBody({
            error: like('Validation failed'),
            code: string('VALIDATION_ERROR'),
            details: eachLike({
              field: like('email'),
              message: like('Must be a valid email address'),
            }),
          });
        })
        .executeTest(async (mockserver) => {
          const userService = new UserService(mockserver.url);

          await expect(
            userService.createUser({
              name: 'No Email',
              email: 'not-an-email',
              role: 'developer',
            })
          ).rejects.toThrow('Validation failed');
        });
    });
  });

  // ── Pact file is written after all tests pass ──────────────
  afterAll(async () => {
    await provider.writePact();
  });
});
```

### 3.3 Provider Verification

```typescript

// provider/tests/user-service.pact-verify.spec.ts
// Provider verification: Can the API fulfill the consumer's contract?
// ============================================================
import { VerifierV4, type V4Verifier } from '@pact-foundation/pact';
import path from 'node:path';

// Your real Express/Fastify/Koa server
import { createApp } from '../src/app';
import type { Server } from 'node:http';

describe('UserService Provider Verification', () => {
  let server: Server;

  beforeAll(async () => {
    const app = await createApp({
      database: {
        // Use a test database with seeded data
        connectionString: process.env.TEST_DATABASE_URL || 'postgres://localhost:5432/user_service_test',
      },
    });

    server = app.listen(0); // Random port
  });

  afterAll(() => {
    server.close();
  });

  it('validates the expectations of the web-app consumer', async () => {
    const verifier: V4Verifier = VerifierV4.create({
      provider: 'user-service',
      providerBaseUrl: `http://localhost:${(server.address() as any).port}`,
      pactUrls: [
        // Path to the pact file generated by consumer tests
        path.resolve(__dirname, '../../consumer/pacts/web-app-user-service.json'),
      ],
      // ── Provider State Setup ─────────────────────────────
      // Each `given()` in consumer tests maps to a state handler here
      publishVerificationResult: true,
      providerVersion: process.env.GIT_SHA || '0.0.0',
      // Enable pending pacts (new interactions not yet verified)
      enablePending: false,
    });

    await verifier.verify();
  });
});
```

## 4. Property-Based Testing with fast-check

### 4.1 Fundamentals: Properties vs. Examples

Traditional example-based testing verifies specific inputs:

```typescript
// Example-based: You must guess the edge cases
expect(add(2, 3)).toBe(5);
expect(add(0, 0)).toBe(0);
expect(add(-1, 1)).toBe(0);
```

Property-based testing describes **invariants** that should hold for ALL valid inputs. fast-check generates thousands of random inputs and verifies the property.

```typescript
// Property-based: The invariant always holds
fc.assert(
  fc.property(fc.integer(), fc.integer(), (a, b) => {
    expect(add(a, b)).toBe(a + b); // Commutative? Add catches the bug.
  })
);
```

```typescript
// ============================================================
// src/utils/string-utils.property.spec.ts
// Property-based tests with fast-check + Vitest
// ============================================================
import { test, expect } from 'vitest';
import { fc, test as fcTest } from '@fast-check/vitest';

import {
  slugify,
  truncate,
  capitalize,
  isValidEmail,
} from './string-utils';

// ── Using @fast-check/vitest decorator ──────────────────────
fcTest.prop([fc.string({ minLength: 1 })])(
  'slugify: result contains no uppercase or special chars',
  (input) => {
    const result = slugify(input);

    // Property: result only contains lowercase, hyphens, and digits
    expect(result).toMatch(/^[a-z0-9-]+$/);

    // Property: result is never longer than input (hyphens ≤ spaces)
    expect(result.length).toBeLessThanOrEqual(input.length + 10);
  }
);

fcTest.prop([fc.string({ minLength: 1, maxLength: 100 }), fc.nat({ max: 50 })])(
  'truncate: result length does not exceed max',
  (input, maxLength) => {
    const result = truncate(input, maxLength);

    // Core property
    expect(result.length).toBeLessThanOrEqual(maxLength);

    // If we didn't truncate, the result equals the input
    if (input.length <= maxLength) {
      expect(result).toBe(input);
    }
  }
);

fcTest.prop([fc.string()])(
  'capitalize: first character is uppercase, rest unchanged in case',
  (input) => {
    const result = capitalize(input);

    if (input.length === 0) {
      expect(result).toBe('');
      return;
    }

    // Property: first char uppercased
    expect(result[0]).toBe(input[0].toUpperCase());

    // Property: rest of string unchanged (including case)
    expect(result.slice(1)).toBe(input.slice(1));
  }
);

// ── Email validation with custom constraints ─────────────────
fcTest.prop([
  fc.record({
    local: fc.string({ minLength: 1, maxLength: 64 }).filter(
      (s) => /^[a-zA-Z0-9.!#$%&'*+/=?^_`{|}~-]+$/.test(s) && !s.startsWith('.') && !s.endsWith('.'))
    ,
    domain: fc.string({ minLength: 1, maxLength: 253 }).filter(
      (s) => s.includes('.') && !s.startsWith('.') && !s.startsWith('-')
    ),
  }),
])(
  'isValidEmail: accepts RFC 5322-compliant emails',
  ({ local, domain }) => {
    const email = `${local}@${domain}`;
    expect(isValidEmail(email)).toBe(true);
  }
);

fcTest.prop([fc.string()])(
  'isValidEmail: rejects random strings',
  (input) => {
    // Precondition: the string doesn't look like an email
    fc.pre(!input.includes('@') || input.split('@').length > 2);

    expect(isValidEmail(input)).toBe(false);
  }
);
```

### 4.3 Custom Arbitraries & Model-Based Testing

Model-based testing verifies that a system's behavior is consistent across sequences of operations.

```typescript
// ============================================================
// src/models/shopping-cart.model.spec.ts
// Model-based testing: Shopping Cart state machine
// ============================================================
import { test, expect } from 'vitest';
import fc from 'fast-check';

import { ShoppingCart, type CartItem } from './shopping-cart';

// ── Define the model ─────────────────────────────────────────
interface CartModel {
  items: Map<string, { product: CartItem; quantity: number }>;
}

// Available commands (operations on the cart)
type CartCommand =
  | { type: 'AddItem'; item: CartItem; quantity: number }
  | { type: 'RemoveItem'; productId: string }
  | { type: 'UpdateQuantity'; productId: string; quantity: number }
  | { type: 'ClearCart' };

// ── Command Arbitrary ────────────────────────────────────────
const productArbitrary = fc.record<CartItem>({
  id: fc.uuid(),
  name: fc.string({ minLength: 1, maxLength: 50 }),
  price: fc.float({ min: 0.01, max: 9999.99, noDefaultInfinity: true, noNaN: true }),
});

const commandArbitrary = (model: CartModel): fc.Arbitrary<CartCommand> => {
  const existingIds = [...model.items.keys()];

  const commands: fc.Arbitrary<CartCommand>[] = [
    // Always can add items
    fc.record({
      type: fc.constant('AddItem' as const),
      item: productArbitrary,
      quantity: fc.integer({ min: 1, max: 10 }),
    }),
    // Always can clear
    fc.constant({ type: 'ClearCart' as const }),
  ];

  // Only can remove/update if there are items
  if (existingIds.length > 0) {
    const existingId = fc.constantFrom(...existingIds);

    commands.push(
      fc.record({
        type: fc.constant('RemoveItem' as const),
        productId: existingId,
      }),
      fc.record({
        type: fc.constant('UpdateQuantity' as const),
        productId: existingId,
        quantity: fc.integer({ min: 1, max: 10 }),
      })
    );
  }

  return fc.oneof(...commands);
};

// ── Model-Based Test ─────────────────────────────────────────
test('ShoppingCart: model-based property test', () => {
  fc.assert(
    fc.property(
      fc.commands(
        [productArbitrary],      // Arbitraries used in commands
        { maxCommands: 50 }
      ),
      (cmds) => {
        // Create real cart and model
        const cart = new ShoppingCart();
        const model: CartModel = { items: new Map() };

        // Execute commands against both
        fc.modelRun(() => ({ model, real: cart }), cmds);
      }
    ),
    { numRuns: 100 } // Run 100 sequences of up to 50 commands each
  );
});

// ── Manual command execution ─────────────────────────────────
// (Alternative without fc.commands helper — more explicit)

function executeCommand(model: CartModel, cart: ShoppingCart, command: CartCommand): void {
  switch (command.type) {
    case 'AddItem': {
      model.items.set(command.item.id, {
        product: command.item,
        quantity: (model.items.get(command.item.id)?.quantity ?? 0) + command.quantity,
      });
      cart.addItem(command.item, command.quantity);
      break;
    }
    case 'RemoveItem': {
      model.items.delete(command.productId);
      cart.removeItem(command.productId);
      break;
    }
    case 'UpdateQuantity': {
      const existing = model.items.get(command.productId);
      if (existing) {
        existing.quantity = command.quantity;
      }
      cart.updateQuantity(command.productId, command.quantity);
      break;
    }
    case 'ClearCart': {
      model.items.clear();
      cart.clear();
      break;
    }
  }

  // ── Assert model and real system match after each command ──
  // Total items
  const modelTotalItems = [...model.items.values()].reduce((sum, v) => sum + v.quantity, 0);
  expect(cart.totalItems).toBe(modelTotalItems);

  // Subtotal
  const modelSubtotal = [...model.items.values()].reduce(
    (sum, v) => sum + v.product.price * v.quantity,
    0
  );
  expect(cart.subtotal).toBeCloseTo(modelSubtotal, 2);

  // Item count matches
  expect(cart.items.length).toBe(model.items.size);
}
```

### 4.4 Shrinking & Reproduction

When fast-check finds a failing input, it **shrinks** it to the minimal counterexample:

```typescript
// ============================================================
// Example: fast-check shrink output
// ============================================================

// Property: Sorting should keep the first element as min
fc.assert(
  fc.property(fc.array(fc.integer()), (arr) => {
    const sorted = [...arr].sort((a, b) => a - b);
    if (arr.length > 0) {
      // BUG: This property is false for negative numbers
      expect(sorted[0]).toBe(Math.max(...arr)); // WRONG — should be Math.min
    }
  })
);

// fast-check reports:
// ✗ Property failed after 3 tests
// { seed: -1604898728, path: "2:0:0:0:0:0:0:0", endOnFailure: true }
// Counterexample: [[0, -1]]   ← SHRUNK to minimal case
// Shrunk 8 time(s)
// Got error: expect(received).toBe(expected)
//   Expected: -1
//   Received: 0           ← sorted[0] is 0, not the min (-1)
```


#### Generating a Test Report

```typescript
// ============================================================
// src/utils/report-property-results.ts
// Custom reporter for fast-check in CI
// ============================================================
import fc from 'fast-check';
import fs from 'node:fs';
import path from 'node:path';

interface PropertyResult {
  name: string;
  passed: boolean;
  numRuns: number;
  numSkips: number;
  counterexample?: unknown;
  error?: string;
  seed?: number;
  duration: number;
}

const results: PropertyResult[] = [];

export function recordPropertyResult(result: PropertyResult): void {
  results.push(result);
}

export function writePropertyReport(): void {
  const report = {
    timestamp: new Date().toISOString(),
    totalProperties: results.length,
    passed: results.filter((r) => r.passed).length,
    failed: results.filter((r) => !r.passed).length,
    results,
  };

  const reportDir = path.resolve(__dirname, '../../property-test-results');
  fs.mkdirSync(reportDir, { recursive: true });
  fs.writeFileSync(
    path.join(reportDir, 'property-report.json'),
    JSON.stringify(report, null, 2)
  );

  if (report.failed > 0) {
    console.error(`\n❌ ${report.failed} property test(s) failed. See report for counterexamples.`);
    process.exitCode = 1;
  }
}
```

## 5. Snapshot Strategies

### 5.1 Inline vs. External Snapshots

Vitest supports two snapshot formats. Choosing the right one impacts PR reviewability and maintenance.

#### Inline Snapshots (`toMatchInlineSnapshot()`)

```typescript
// ============================================================
// src/utils/formatters.snap.spec.ts
// Inline snapshots — good for small, stable outputs
// ============================================================
import { describe, it, expect } from 'vitest';
import { formatUser, type User } from './formatters';

describe('formatUser', () => {
  it('formats a complete user profile (inline)', () => {
    const user: User = {
      id: 'usr_1',
      name: 'Grace Hopper',
      email: 'grace@navy.mil',
      role: 'admin',
    };

    expect(formatUser(user)).toMatchInlineSnapshot(`
      "User: Grace Hopper
      Email: grace@navy.mil
      Role: admin"
    `);
  });
});
```

**Inline snapshot pros/cons:**
- ✅ Visible in PR diff — easy to review
- ✅ No external files to manage
- ❌ Clutters test files for large outputs
- ❌ Hard to find all snapshots in a codebase

#### External Snapshots (`toMatchSnapshot()`)

```typescript
// ============================================================
// src/utils/api-responses.snap.spec.ts
// External snapshots — good for large or frequently-updated outputs
// ============================================================
import { describe, it, expect } from 'vitest';
import { serializeSearchResults, type SearchResult } from './api-serializers';

describe('serializeSearchResults', () => {
  it('serializes a multi-page API response', () => {
    const results: SearchResult[] = [
      { id: '1', title: 'TypeScript Handbook', score: 0.95 },
      { id: '2', title: 'React Docs', score: 0.87 },
      { id: '3', title: 'Node.js Guide', score: 0.72 },
    ];

    // Stored in: src/utils/__snapshots__/api-responses.snap.spec.ts.snap
    expect(serializeSearchResults(results)).toMatchSnapshot();
  });

  it('serializes empty results', () => {
    expect(serializeSearchResults([])).toMatchSnapshot();
  });
});
```

### 5.2 Custom Serializers

Not everything serializes well by default. Custom serializers normalize unstable data.

```typescript
// ============================================================
// tests/serializers.ts — Custom snapshot serializers
// ============================================================
import { expect } from 'vitest';

// ── 1. Strip unstable fields ─────────────────────────────────
expect.addSnapshotSerializer({
  serialize(val: any, config, indentation, depth, refs, printer) {
    // Clone and remove timestamp fields
    const cleaned = JSON.parse(JSON.stringify(val));

    function stripTimestamps(obj: any): void {
      if (!obj || typeof obj !== 'object') return;
      for (const key of Object.keys(obj)) {
        if (key === 'createdAt' || key === 'updatedAt' || key === 'timestamp') {
          obj[key] = '[TIMESTAMP]';
        } else if (typeof obj[key] === 'object') {
          stripTimestamps(obj[key]);
        }
      }
    }

    stripTimestamps(cleaned);
    return printer(cleaned, config, indentation, depth, refs);
  },

  test(val: any) {
    return (
      typeof val === 'object' &&
      val !== null &&
      ('createdAt' in val || 'updatedAt' in val || 'timestamp' in val)
    );
  },
});

// ── 2. Normalize IDs ─────────────────────────────────────────
expect.addSnapshotSerializer({
  serialize(val: any, config, indentation, depth, refs, printer) {
    const idMap = new Map<string, string>();
    let counter = 1;

    function normalizeIds(obj: any): any {
      if (typeof obj === 'string' && /^[a-z]+_[a-z0-9]+$/i.test(obj)) {
        if (!idMap.has(obj)) {
          idMap.set(obj, `<ID:${counter++}>`);
        }
        return idMap.get(obj);
      }
      if (Array.isArray(obj)) return obj.map(normalizeIds);
      if (obj && typeof obj === 'object') {
        const result: Record<string, any> = {};
        for (const [key, value] of Object.entries(obj)) {
          result[key] = normalizeIds(value);
        }
        return result;
      }
      return obj;
    }

    return printer(normalizeIds(val), config, indentation, depth, refs);
  },

  test(val: any) {
    if (typeof val !== 'object' || val === null) return false;
    const str = JSON.stringify(val);
    return /[a-z]+_[a-z0-9]{10,}/i.test(str);
  },
});

// ── 3. Truncate long strings ─────────────────────────────────
expect.addSnapshotSerializer({
  serialize(val: string, config, indentation, depth, refs, printer) {
    if (val.length > 80) {
      return printer(
        `${val.slice(0, 77)}...`,
        config,
        indentation,
        depth,
        refs
      );
    }
    return printer(val, config, indentation, depth, refs);
  },

  test(val: any) {
    return typeof val === 'string' && val.length > 80;
  },
});
```

### 5.4 Large Snapshot Management

For large snapshots (API responses, big component trees), use these strategies to keep them manageable.

#### Strategy 1: Snapshot Subsets

```typescript
// ============================================================
// Instead of snapshotting the entire object, snapshot key parts
// ============================================================
import { describe, it, expect } from 'vitest';

describe('Large API Response', () => {
  it('has correct structure', async () => {
    const response = await fetch('/api/large-dataset');
    const data = await response.json();

    // ❌ Don't do this for 1000+ items:
    // expect(data).toMatchSnapshot();

    // ✅ Snapshot the structure (first 3 items + metadata):
    expect({
      total: data.total,
      pageSize: data.pageSize,
      firstThree: data.items.slice(0, 3),
      lastThree: data.items.slice(-3),
    }).toMatchSnapshot();
  });
});
```

#### Strategy 2: Snapshot File Splitting

```typescript
// ============================================================
// vitest.config.ts — Split large snapshots into manageable files
// ============================================================
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    // Name snapshots with the test file path to avoid collisions
    snapshotFormat: {
      escapeString: false,
      printBasicPrototype: false,
    },
    // Custom snapshot resolver: colocate snapshots with tests
    resolveSnapshotPath: (testPath, snapExtension) => {
      // src/components/Button.test.tsx → src/components/__snapshots__/Button.test.tsx.snap
      const dir = testPath.replace(/\/[^/]+$/, '/__snapshots__');
      const file = testPath.split('/').pop()!;
      return `${dir}/${file}${snapExtension}`;
    },
  },
});
```

#### Strategy 3: Snapshot Thresholds & Monitoring

```typescript
// ============================================================
// scripts/check-snapshot-sizes.ts
// CI script to warn when snapshots grow too large
// ============================================================
import fs from 'node:fs';
import path from 'node:path';
import { glob } from 'glob';

const SNAPSHOT_SIZE_WARN_KB = 50;
const SNAPSHOT_SIZE_ERROR_KB = 200;

async function checkSnapshotSizes(): Promise<void> {
  const snapFiles = await glob('**/__snapshots__/**/*.snap', {
    ignore: ['node_modules/**'],
  });

  let hasWarnings = false;
  let hasErrors = false;

  for (const file of snapFiles) {
    const stat = fs.statSync(file);
    const sizeKB = stat.size / 1024;

    if (sizeKB > SNAPSHOT_SIZE_ERROR_KB) {
      console.error(
        `❌ ERROR: Snapshot too large: ${file} (${sizeKB.toFixed(1)} KB). ` +
        `Max allowed: ${SNAPSHOT_SIZE_ERROR_KB} KB. Consider splitting or using matchers.`
      );
      hasErrors = true;
    } else if (sizeKB > SNAPSHOT_SIZE_WARN_KB) {
      console.warn(
        `⚠️  WARNING: Snapshot approaching limit: ${file} (${sizeKB.toFixed(1)} KB)`
      );
      hasWarnings = true;
    }
  }

  if (hasErrors) {
    process.exit(1);
  }
  if (hasWarnings) {
    console.warn('\n⚠️  Some snapshots are large. Review before they become unmanageable.');
  }
}

checkSnapshotSizes();
```

### 5.5 Snapshot Strategy Decision Matrix

| Scenario | Strategy | Rationale |
|---|---|---|
| Small config objects (< 20 lines) | **Inline snapshots** | Easy to review in PR diff |
| API response with dynamic IDs/timestamps | **Property matchers** (`expect.any()`, `expect.stringMatching()`) | Avoids flake from unstable values |
| Large API response (100+ items) | **Subset snapshots** (first/last N items) | Keeps snapshot files manageable |
| Rendered component tree | **Inline snapshots** or **external with custom serializer** | Capture DOM structure without implementation details |
| Error messages (static) | **Inline snapshots** | Messages change rarely; inline makes changes obvious |
| Generated code output | **External snapshots** + CI size check | Code can be large; external keeps tests clean |
| Objects with `Date`, `Map`, `Set` | **Custom serializer** | Native serialization is lossy; serialize to stable format |
| Frequently changing data | **Property matchers** or skip snapshot | If it changes every commit, snapshots create noise |
| E2E visual snapshots | **Playwright `toHaveScreenshot()`** | Real browser rendering, not text |
| Design system components | **Chromatic/Percy** | Visual review workflow, cross-browser |
| API contracts | **Pact contract files** | Consumer-driven, verifiable by provider |
