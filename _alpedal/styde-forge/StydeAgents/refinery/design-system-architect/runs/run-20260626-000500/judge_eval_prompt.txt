## 1. Design Principles

| Principle | Description |
|-----------|-------------|
| **Clarity** | Interfaces must be immediately understandable. Use familiar patterns and clear visual hierarchy. |
| **Consistency** | Every component shares a single source of truth (design tokens). No one-off values. |
| **Efficiency** | Minimize cognitive load. Progressive disclosure. Shortest path to completion. |
| **Accessibility** | WCAG 2.2 AA compliance minimum. All components must be keyboard-navigable and screen-reader friendly. |
| **Resilience** | Components degrade gracefully. Handle empty, loading, error, and extreme-content states. |
| **Theme-Agnostic** | Every component must render correctly in both light and dark themes without layout shifts. |
| **Touch-First** | All interactive elements meet mobile touch target requirements by default. Desktop interactions are additive, never subtractive. |
| **Composability** | Components are building blocks. Complex UIs emerge from composition, not monolithic configuration. |

| Requirement | Implementation |
|-------------|----------------|

## 3. Theming Strategy

### 3.1 Light Theme

The default theme. Activated when `data-theme="light"` or no `data-theme` set on `<html>`.

**Characteristics:**
- High contrast ratios (minimum 4.5:1 body, 3:1 large text)
- Bright backgrounds with dark text
- Subtle gray borders and shadows
- Blue-600 brand accents

**Key Background Stack:**

```
Page bg:        gray-50  (#f9fafb)
Card bg:        white    (#ffffff)
Input bg:       white    (#ffffff)
Elevated bg:    white    (#ffffff)
```

### 3.2 Dark Theme

Activated when `data-theme="dark"` on `<html>`.

**Characteristics:**
- Dark backgrounds (`#030712` page, `#111827` card)
- Light text (`#f9fafb` primary)
- Softer borders (`gray-700`)
- Deeper shadows with higher opacity (same token names, different values)
- Brand colors maintain saturation; blue-400 for links

**Key Background Stack:**

```
Page bg:        gray-950 (#030712)
Card bg:        gray-900 (#111827)
Input bg:       gray-800 (#1f2937)
Elevated bg:    gray-800 (#1f2937)
```

**Color Adjustments in Dark Mode:**
- Gray palette is inverted (50 ↔ 950, 100 ↔ 900, etc.)
- Blue, red, green, yellow mid-tones (500) remain unchanged for brand recognition
- Semantic bg tokens switch to darker variants
- Text tokens invert: `text-primary` maps to `gray-50` instead of `gray-900`
- Shadow tokens are replaced with higher-opacity variants (same token names)
- Data visualization tokens swap to their dark variants

### 3.3 Theme Switching Mechanism

```
Implementation Strategy: CSS Custom Properties + data-theme attribute
```

**Architecture:**

1. **Token Layer (CSS):** All components reference `--sty-*` tokens exclusively — never hardcoded hex values.
2. **Theme Stylesheet:** Two blocks: `:root, [data-theme="light"]` and `[data-theme="dark"]` defining the same tokens with different primitive references.
3. **Runtime Switching:**
   ```js
   // Set theme
   document.documentElement.setAttribute('data-theme', 'dark');
   // Persist
   localStorage.setItem('sty-theme', 'dark');
   // Respect OS preference on first visit
   const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
   const stored = localStorage.getItem('sty-theme');
   const theme = stored ?? (prefersDark ? 'dark' : 'light');
   ```
4. **Transition:** Apply a brief `transition: background-color 200ms ease, color 200ms ease` on `:root` to avoid jarring flashes.
5. **System Sync:** Listen for `matchMedia` changes to update theme when OS preference changes (if user hasn't manually set a preference).


**Composite pattern:**

### 4.12 Skeleton / Loading System

A comprehensive system for representing loading states across all components. The Skeleton system provides atomic building blocks and higher-level composition patterns.

#### 4.12.1 Core Skeleton Component

```typescript
interface SkeletonProps {
  /** Shape variant */
  variant?: 'text' | 'circular' | 'rectangular';
  /** Width */
  width?: string | number;
  /** Height */
  height?: string | number;
  /** Animation */
  animation?: 'pulse' | 'wave' | 'none';
  /** Speed */
  speed?: 'slow' | 'normal' | 'fast';
  /** Custom class */
  className?: string;
}
```

| `variant` | Default Size | Border Radius | Usage |
|-----------|-------------|---------------|-------|
| `text` | `100% × 1em` | `--sty-radius-sm` | Simulates lines of text |
| `circular` | `40px × 40px` | `--sty-radius-full` | Avatars, icons, status dots |
| `rectangular` | `100% × 100px` | `--sty-radius-md` | Images, cards, charts, buttons |

| `animation` | Description | CSS |
|-------------|-------------|-----|
| `pulse` | Opacity oscillates between 0.4 and 0.7 | `animation: skeleton-pulse 1.5s ease-in-out infinite` |
| `wave` | Shimmer gradient sweeps left to right | `background: linear-gradient(90deg, var(--sty-color-bg-tertiary) 25%, var(--sty-color-bg-secondary) 50%, var(--sty-color-bg-tertiary) 75%); background-size: 200% 100%; animation: skeleton-wave 1.5s ease-in-out infinite` |
| `none` | Static placeholder, no animation | Static `--sty-color-bg-tertiary` fill |

```css
@keyframes skeleton-pulse {
  0%, 100% { opacity: 0.4; }
  50%      { opacity: 0.7; }
}

@keyframes skeleton-wave {
  0%   { background-position: 200% 0; }
  100% { background-position: -200% 0; }
}
```

| `speed` | Duration |
|---------|----------|
| `slow` | `2.0s` |
| `normal` | `1.5s` |
| `fast` | `1.0s` |

```
┌──────────┬──────────────┬──────────┬──────────┐
│ ████████ │ ████████████ │ ████████ │ ████████ │ ← header skeleton
├──────────┼──────────────┼──────────┼──────────┤
```


#### 4.12.4 Skeleton Best Practices

| Do | Don't |
|----|-------|
| Match skeleton dimensions to real content | Use generic, same-size skeletons everywhere |
| Use `SkeletonGroup` with `loading` prop to swap between skeleton and real content | Manually toggle visibility with conditional rendering |
| Animate skeletons (`pulse` or `wave`) so users know content is loading | Use static gray boxes that look like missing images |
| Show skeleton immediately on navigation, not after a delay | Show a blank white page while data fetches |
| Use skeletons for page/section-level loading | Use skeletons for sub-100ms operations (feel like jank) |
| Respect `prefers-reduced-motion` — disable animation | Animate skeletons for users with motion sensitivity |

#### 4.12.5 Loading Strategy Decision Tree

```
                    ┌─────────────────┐
                    │ Content loading? │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
         First load?    Action/Button?   Refetch?
              │              │              │
              ▼              ▼              ▼
        Skeleton         Spinner       Stale content
        screen           in button     (no loading UI)
        (full page)      (inline)      or subtle
              │              │         indicator
              ▼              ▼
         Use layout-    Disable button,
         aware          show spinner
         skeleton       left of text
         patterns
```

**Rule of thumb:**
- **< 200ms**: No loading indicator needed (feel instantaneous)
- **200ms – 1s**: Inline skeleton or spinner on the specific element
- **1s – 5s**: Full skeleton screen matching the final layout
- **> 5s**: Skeleton + progress indicator or estimated time message

## 5. Component Usage Guidelines

### 5.1 Composition Patterns

**Rule: Prefer composition over configuration.** Components expose slot props (`header`, `footer`, `leftElement`, `rightElement`) rather than bloated prop surfaces.

**✅ DO:**
```jsx
<Card header={<CardTitle>Revenue</CardTitle>}>
  <Chart data={revenueData} />
</Card>
```

**❌ DON'T:**
```jsx
<Card title="Revenue" chartData={revenueData} />
```

#### 5.1.1 Core Composition Patterns

**Pattern 1: Slots over Props**

Components use named slots for extensibility. A slot is a prop that accepts `ReactNode`, allowing any element to be injected.

```
Component        Slots
─────────        ─────
Button           leftIcon, rightIcon, children
Card             header, footer, children
Input            leftElement, rightElement
Modal            header (via title), footer, children
Table            emptyState, errorState, footer, columns[].render
Badge            icon, children (label)
Tabs             Tab (children), TabPanel (children)
Dropdown         trigger, DropdownItem, DropdownHeader, DropdownDivider
Select           options, or children with <Select.Option>
Accordion        AccordionItem (trigger, children)
```

**Pattern 2: Compound Components**

Related sub-components are exposed as static properties on the parent:

```jsx
// Dropdown
<Dropdown trigger={<Button>Menu</Button>}>
  <Dropdown.Item icon={<EditIcon />}>Edit</Dropdown.Item>
  <Dropdown.Divider />
  <Dropdown.Item destructive>Delete</Dropdown.Item>
</Dropdown>

// Tabs
<Tabs value={tab} onChange={setTab}>
  <Tabs.List>
    <Tabs.Tab value="general">General</Tabs.Tab>
    <Tabs.Tab value="billing">Billing</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panels>
    <Tabs.Panel value="general"><GeneralSettings /></Tabs.Panel>
    <Tabs.Panel value="billing"><BillingSettings /></Tabs.Panel>
  </Tabs.Panels>
</Tabs>

// Select (compound variant)
<Select value={selected} onChange={setSelected}>
  <Select.Option value="apple">Apple</Select.Option>
  <Select.Option value="banana" disabled>Banana</Select.Option>
  <Select.Option value="cherry">Cherry</Select.Option>
</Select>

// Accordion
<Accordion multiple value={open} onChange={setOpen}>
  <Accordion.Item value="faq-1" trigger="What is Stryde?">
    Stryde is a design system for modern web applications.
  </Accordion.Item>
  <Accordion.Item value="faq-2" trigger="How do I get started?">
    Install the package and import the components you need.
  </Accordion.Item>
</Accordion>

// Pagination (compound variant)
<Pagination page={page} totalPages={20} onChange={setPage}>
  <Pagination.First />
  <Pagination.Prev />
  <Pagination.Pages />
  <Pagination.Next />
  <Pagination.Last />
</Pagination>
```

**Pattern 3: Controlled & Uncontrolled**

Every stateful component supports both modes:

```jsx
// Controlled — parent manages state
<Modal isOpen={isOpen} onClose={() => setIsOpen(false)} />

// Uncontrolled — component manages its own state
<Dropdown defaultIsOpen={false} trigger={<Button>Menu</Button>}>
  <DropdownItem>Action</DropdownItem>
</Dropdown>

// Accordion with both modes
<Accordion value={openItems} onChange={setOpenItems}> {/* controlled */}
<Accordion defaultValue={['faq-1']}>                        {/* uncontrolled */}
```

**Pattern 4: Polymorphic Components**

Components support an `as` prop to change the underlying HTML element:

```jsx
<Button as="a" href="/dashboard">Go to Dashboard</Button>
<Card as="section" aria-label="User profile">...</Card>
<Badge as="span">New</Badge>
```

**Pattern 5: Render Props for Customization**

Where slots aren't enough, components accept render functions:

```jsx
<Table
  data={users}
  columns={[
    {
      key: 'status',
      header: 'Status',
      render: (user) => <Badge variant={user.status === 'active' ? 'success' : 'error'} label={user.status} />
    }
  ]}
/>
```

**Pattern 6: Composing Loading States**

Loading states compose naturally using the skeleton system:

```jsx
// Option A: Pass loading prop
<Card loading={isLoading}>
  <UserProfile user={user} />
</Card>

// Option B: Use SkeletonGroup for custom layouts
<SkeletonGroup loading={isLoading} spacing="12px">
  <UserProfile user={user} />
</SkeletonGroup>

// Option C: Manual composition for full control
{isLoading ? <UserProfileSkeleton /> : <UserProfile user={user} />}
```

**Pattern 7: Layout Primitives (Convenience Exports)**

```typescript
interface StackProps {
  /** Spacing between children */
  spacing?: keyof typeof spacingTokens;
  /** Direction */
  direction?: 'vertical' | 'horizontal';
  /** Alignment */
  align?: 'start' | 'center' | 'end' | 'stretch';
  /** Justification */
  justify?: 'start' | 'center' | 'end' | 'between' | 'around';
  /** Wrap children */
  wrap?: boolean;
  /** Children */
  children: ReactNode;
}

interface GridProps {
  /** Number of columns */
  columns?: number | Record<string, number>; // responsive: { base: 1, md: 2, lg: 3 }
  /** Gap between items */
  gap?: keyof typeof spacingTokens;
  /** Children */
  children: ReactNode;
}
```

```jsx
<Stack direction="horizontal" spacing="4" align="center">
  <Avatar src={user.avatar} />
  <Stack spacing="1">
    <Text variant="body">{user.name}</Text>
    <Text variant="caption">{user.email}</Text>
  </Stack>
</Stack>

<Grid columns={{ base: 1, md: 2, lg: 3 }} gap="6">
  {cards.map(card => <Card key={card.id}>...</Card>)}
</Grid>
```

**Anti-patterns to Avoid:**

| Anti-pattern | Why it's bad | Do this instead |
|-------------|--------------|-----------------|
| Over-configuration (50+ props) | Hard to document, test, and compose | Split into slots, compound components, or sub-components |
| Implicit state coupling | Components with hidden dependencies break in isolation | Use controlled props + clear data flow |
| Prop drilling through many layers | Fragile, hard to refactor | Use Context providers where appropriate (ToastProvider, ThemeProvider) |
| Mixing composition and configuration | Inconsistent API surface | Pick one primary pattern per component |
| Div soup without semantic wrappers | Inaccessible and hard to style | Use Card, Stack, Grid primitives |

### 5.2 Accessibility Requirements

All components must meet these baseline requirements:

| Requirement | Implementation |
|-------------|----------------|
| **Keyboard navigation** | Tab/Shift+Tab for focus order. Arrow keys for composite widgets (tabs, dropdown, table sort). Enter/Space to activate. Escape to dismiss overlays. |
| **Focus management** | Modals/Dropdowns trap focus. Focus returns to trigger on close. Focus rings visible (`outline: 2px solid --sty-color-border-focus` with `2px` offset). |
| **ARIA attributes** | All interactive elements have appropriate roles, labels, and states. Generated IDs for `aria-labelledby` / `aria-describedby` relationships. |
| **Screen reader announcements** | Dynamic content changes (toast, modal open, loading complete) use `aria-live` regions (`polite` for info, `assertive` for errors). |
| **Color contrast** | Minimum 4.5:1 for normal text, 3:1 for large text (18px+ bold or 24px+ regular), 3:1 for UI components and graphical objects. |
| **Reduced motion** | Respect `prefers-reduced-motion: reduce`. Disable animations, use instant transitions. |
| **Touch targets** | Minimum 44×44px interactive area (WCAG 2.5.5). See [§5.6 Mobile Touch Targets Specification](#56-mobile-touch-targets-specification). |
| **Focus indication** | All interactive elements must have a visible focus indicator. See [§5.7 Focus-Visible Style Guide](#57-focus-visible-style-guide). |

### 5.3 Responsive Behavior

**Mobile-first approach:** Base styles target small viewports. Breakpoints add overrides for larger screens.

| Component | Mobile Behavior |
|-----------|----------------|
| **Modal** | Fullscreen at `sm` breakpoint, centered dialog above |
| **Table** | Horizontal scroll wrapper at `sm`; optional card layout (each row becomes a card) at `xs` |
| **Tabs** | Horizontal scroll if tabs overflow; `overflow="scroll"` |
| **Dropdown** | On narrow viewports, may render as a bottom sheet |
| **Toast** | Full-width at `sm`, positioned blocks above |
| **Card** | Stacks vertically in grids; `--sty-container-*` constrains max-width |
| **Select** | Native `<select>` fallback on touch devices when `searchable` introduces excessive complexity |
| **Pagination** | Compact mode: hide First/Last buttons, reduce to Prev/Page/Next at `sm`; icon-only buttons at `xs` |
| **Accordion** | Always full-width; content padding adjusts by breakpoint |

**Breakpoint Usage in Components:**

```css
/* Base: mobile */
.card-grid { display: grid; grid-template-columns: 1fr; gap: var(--sty-space-4); }

/* Tablet + */
@media (min-width: 768px) {
  .card-grid { grid-template-columns: repeat(2, 1fr); }
}

/* Desktop + */
@media (min-width: 1024px) {
  .card-grid { grid-template-columns: repeat(3, 1fr); }
}
```

### 5.4 State Management

Every interactive component must handle these UI states explicitly:

| State | Definition | Example |
|-------|-----------|---------|
| **Default** | Normal resting state | Button with label |
| **Hover** | Pointer over element | Button background shifts |
| **Focus** | Keyboard focused | Focus ring appears |
| **Focus-Visible** | Keyboard-focused, not pointer-focused | Focus ring only for keyboard users |
| **Active/Pressed** | Being clicked/pressed | Slightly darker background |
| **Disabled** | Non-interactive | Reduced opacity, `cursor: not-allowed` |
| **Loading** | Operation in progress | Spinner, skeleton, or progress bar |
| **Empty** | No data to display | Empty state illustration + message |
| **Error** | Operation failed | Error message + retry action |
| **Selected/Active** | Currently chosen item | Highlighted row, active tab indicator |
| **Expanded** | Expanded accordion/dropdown | Rotated chevron |
| **Indeterminate** | Mixed state (checkbox) | Dash in checkbox |

**Loading States Priority:**

1. **Skeleton screens** for initial page/component load (see §4.12)
2. **Inline spinners** for button actions (submit, save)
3. **Progress bars** for multi-step processes
4. **Optimistic UI** where safe (toggle switches, checkbox states)

### 5.6 Mobile Touch Targets Specification

All interactive elements must meet minimum touch target sizes to comply with WCAG 2.5.5 (Target Size) and provide a usable experience on touch devices.

#### 5.6.1 Core Requirements

| Requirement | Value | Source |
|-------------|-------|--------|
| **Minimum touch target** | 44 × 44 CSS pixels | WCAG 2.5.5 (AA) |
| **Minimum spacing between targets** | 8px (center-to-center when possible) | Best practice |
| **Inline link exception** | Allowed if equivalent 44px target exists elsewhere | WCAG 2.5.5 |

#### 5.6.2 Per-Component Touch Target Dimensions

| Component | Default Touch Area | Strategy |
|-----------|-------------------|----------|
| **Button (xs)** | 28 × 28px visual, **44 × 28px** hit zone | Padding extends vertically to 44px via transparent hit area or inline padding |
| **Button (sm)** | 32 × 32px visual, **44 × 32px** hit zone | Same as xs |
| **Button (md)** | **44 × 40px** (meets target natively) | No extra padding needed; 40px height + 2px focus ring offset = 44px |
| **Button (lg)** | **48+ × 48px** | Exceeds minimum |
| **Button (xl)** | **56+ × 56px** | Exceeds minimum |
| **Checkbox (sm)** | **44 × 24px** (control is 16px + padding) | Clickable area padded to 44px width |
| **Checkbox (md)** | **44 × 28px** (control is 20px + padding) | Clickable area padded to 44px width |
| **Radio (sm)** | **44 × 24px** | Same strategy as checkbox |
| **Radio (md)** | **44 × 28px** | Same strategy as checkbox |
| **Switch (sm)** | **44 × 24px** (track 32px + label padding) | Entire row is clickable |
| **Switch (md)** | **44 × 28px** (track 44px wide) | Track alone meets 44px width |
| **Input (sm)** | **44px tall** (32px visual + 12px padding/overflow) | Height enforced via min-height |
| **Input (md)** | **44px tall** (40px visual + 4px padding) | Height enforced via min-height |
| **Input (lg)** | **48px tall** | Exceeds minimum |
| **Select trigger** | Same as Input heights | Same strategies |
| **Dropdown item** | **44px tall** (40px visual + 2px padding) | Each menu item is at least 44px tall |
| **Tab (sm)** | **44 × 36px** (36px visual + 8px padding) | Padded to 44px |
| **Tab (md)** | **44 × 44px** (44px visual) | Meets natively |
| **Pagination button** | **44 × 44px** minimum (sm variant: 32px with 12px padding extension) | Padded to 44px |
| **Accordion trigger** | **44px tall** minimum | Full row clickable, min-height enforced |
| **Tooltip trigger** | Inherits from wrapped element | Tooltip itself is not interactive |
| **Toast close button** | **44 × 44px** | Always 44px even if icon is 16px |
| **Badge close (×)** | **44 × 24px** | Padded to 44px wide |

#### 5.6.3 Implementation Techniques

**Technique 1: Invisible Padding Extension**

Use transparent padding or a pseudo-element to extend the hit area without affecting visual layout.

```css
.touch-target-extend {
  position: relative;
  /* Visual size: 32px */
  width: 32px;
  height: 32px;
}

.touch-target-extend::after {
  content: '';
  position: absolute;
  /* Extend to 44px */
  top: -6px;
  right: -6px;
  bottom: -6px;
  left: -6px;
}
```

**Technique 2: Min-Height / Min-Width Enforcement**

```css
.sty-input-sm {
  height: 32px;
  min-height: 44px; /* Enforce touch target */
  padding: 6px 8px; /* Visual 32px, hit area 44px */
}

.sty-dropdown-item {
  min-height: 44px;
  display: flex;
  align-items: center;
}
```

**Technique 3: Full-Row Clickable (For list items, accordions, tabs)**

```css
.sty-accordion-trigger {
  display: flex;
  align-items: center;
  width: 100%;
  min-height: 44px;
  padding: var(--sty-space-3) var(--sty-space-4);
}
```

**Technique 4: Negative Margin + Padding (for tight layouts)**

When items must be visually compact but still touchable:

```css
.compact-toolbar-button {
  padding: 6px;         /* Visual padding */
  margin: -6px;         /* Compensate for layout */
  width: 28px;          /* Visual size */
  height: 28px;
  /* Hit area: 28 + 2*6 = 40px... add more */
}

/* Better approach — use 44px directly: */
.compact-toolbar-button-v2 {
  padding: 8px;
  min-width: 44px;
  min-height: 44px;
}
```

#### 5.6.4 Touch Target Testing Checklist

| Check | Method |
|-------|--------|
| All interactive elements ≥ 44×44px? | DevTools: inspect element, check computed box model (content + padding) |
| Touch targets don't overlap? | Visual inspection at 320px viewport |
| Inline elements (<44px) have nearby equivalent? | Check WCAG 2.5.5 "Equivalent" exception |
| Spinner inside button maintains target? | Verify button doesn't shrink when `loading={true}` |
| Focus rings don't overflow? | Ensure outline-offset doesn't cause overflow/scrollbars |
| iOS Safari tap highlight doesn't clip? | Test on real device |

### 5.7 Focus-Visible Style Guide

Comprehensive guidelines for keyboard focus indication across all components. Focus-visible styles are critical for accessibility — users navigating by keyboard must always know which element is focused.

#### 5.7.1 Core Principle

Use `:focus-visible` instead of `:focus` to avoid showing focus rings on mouse clicks while ensuring keyboard users always see them.

```css
/* ❌ Don't — shows focus ring on all clicks */
.button:focus {
  outline: 2px solid var(--sty-color-border-focus);
}

/* ✅ Do — shows focus ring only for keyboard navigation */
.button:focus-visible {
  outline: 2px solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
}
```

#### 5.7.2 Focus Ring Styles by Component Category

**Category 1: Solid Buttons (primary, danger, brand backgrounds)**

Focus ring appears outside the button to maintain contrast against the colored background.

```css
.solid-button:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
  /* Dark theme: outline-offset may need to be 0 since it's already visible against dark bg */
}
```

**Category 2: Outlined / Ghost / Text Buttons (transparent or light backgrounds)**

Focus ring appears at the element's border edge.

```css
.outlined-button:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: 0; /* Snug against border */
}
```

**Category 3: Input Fields**

Focus uses a glow + border color change instead of a standard outline.

```css
.input:focus-visible {
  border-color: var(--sty-color-border-focus);
  box-shadow:
    0 0 0 var(--sty-focus-ring-glow-spread)
    rgba(59, 130, 246, var(--sty-focus-ring-alpha));
  outline: none; /* No double ring */
}

.input-error:focus-visible {
  border-color: var(--sty-focus-ring-color-error);
  box-shadow:
    0 0 0 var(--sty-focus-ring-glow-spread)
    rgba(239, 68, 68, var(--sty-focus-ring-alpha));
}
```

**Category 4: Checkboxes, Radios, Switches**

Focus ring wraps the entire label area for better visibility.

```css
.checkbox-label:focus-visible,
.radio-label:focus-visible {
  /* Focus ring on the wrapper, not just the control */
}

.checkbox-label:focus-visible .checkbox-control,
.radio-label:focus-visible .radio-control {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
}

.switch:focus-visible .switch-track {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
}
```

**Category 5: Links**

Standard outline, same as ghost buttons.

```css
a:focus-visible,
.link:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
  border-radius: var(--sty-radius-xs);
  text-decoration: underline;
}
```

**Category 6: Composite Widgets (Tabs, Dropdown, Accordion)**

Focus ring on the individual trigger, not the container.

```css
.tab-trigger:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: -2px; /* Inset to avoid clipping in scroll container */
  border-radius: var(--sty-radius-md);
}

.dropdown-item:focus-visible,
.menu-item:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: -2px; /* Inset — items fill the menu width */
  background: var(--sty-color-bg-tertiary);
}

.accordion-trigger:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
  z-index: 1; /* Ensure ring isn't clipped by adjacent items */
}
```

**Category 7: Cards (interactive)**

```css
.card-interactive:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
  border-radius: inherit;
}
```

**Category 8: Table Rows (interactive)**

```css
.table-row-interactive:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: -2px; /* Inset — rows have tight boundaries */
  background: var(--sty-color-bg-tertiary);
}
```

#### 5.7.3 Focus Ring Design Tokens Reference

All focus ring styles reference the tokens defined in [§2.11](#211-focus-visible-tokens):

| Token | Effect |
|-------|--------|
| `--sty-focus-ring-width` | Outline thickness (2px) |
| `--sty-focus-ring-offset` | Space between element and outline (2px) |
| `--sty-focus-ring-color` | Blue-500, visible against light and dark |
| `--sty-focus-ring-alpha` | Glow opacity for input-style focus (0.4) |
| `--sty-focus-ring-glow-spread` | Spread radius for glow effects (3px) |
| `--sty-focus-ring-transition` | Smooth appearance |

#### 5.7.4 Focus Management Rules

| Rule | Description |
|------|-------------|
| **One focus ring at a time** | Only the truly focused element shows a ring. Don't show rings on parent containers. |
| **Focus trap in modals** | When modal opens, focus moves to first focusable element. Tab cycles within modal. |
| **Focus return** | When modal/dropdown closes, focus returns to the trigger element. |
| **Roving tabindex** | For composite widgets (tabs, dropdown items), only the active/focused item has `tabindex="0"`; others have `tabindex="-1"`. |
| **Skip links** | Provide a "Skip to content" link as the first focusable element on every page. |
| **Visible on all themes** | Focus ring must have ≥ 3:1 contrast against all possible backgrounds. |
| **Never `outline: none` without replacement** | Removing default outline requires a custom focus style of equal or better visibility. |
| **Respect forced-colors** | Use `outline` (not `box-shadow`) for focus rings so they render in Windows High Contrast Mode. |

#### 5.7.5 Focus Ring Contrast Matrix

Verify the focus ring color (`#3b82f6`) is visible against all backgrounds:

| Background | Hex | Contrast vs. `#3b82f6` | Passes 3:1? |
|------------|-----|------------------------|-------------|
| White | `#ffffff` | 4.6:1 | ✅ |
| Light gray (page bg) | `#f9fafb` | 4.5:1 | ✅ |
| Medium gray | `#d1d5db` | 3.6:1 | ✅ |
| Dark gray (card dark) | `#1f2937` | 3.3:1 | ✅ |
| Very dark (page dark) | `#030712` | 6.8:1 | ✅ |
| Brand blue | `#2563eb` | 1.3:1 | ❌ — Use `#ffffff` outline on brand backgrounds |

**Exception — Focus on brand-colored elements:** When a primary button has focus, the focus ring should be `white` (`#ffffff`) to contrast against the blue background.

```css
.primary-button:focus-visible {
  outline: var(--sty-focus-ring-width) solid white;
  outline-offset: var(--sty-focus-ring-offset);
}
```

#### 5.7.6 Global Focus Reset

```css
/* Global: remove default focus and replace with custom */
*:focus {
  outline: none;
}

*:focus-visible {
  outline: var(--sty-focus-ring-width) solid var(--sty-focus-ring-color);
  outline-offset: var(--sty-focus-ring-offset);
  transition: var(--sty-focus-ring-transition);
}

/* Remove focus ring for mouse users (handled by :focus-visible) */
*:focus:not(:focus-visible) {
  outline: none;
}

/* High contrast mode: ensure focus is visible */
@media (forced-colors: active) {
  *:focus-visible {
    outline: 2px solid ButtonText;
    outline-offset: 2px;
  }
}

/* Respect reduced motion */
@media (prefers-reduced-motion: reduce) {
  *:focus-visible {
    transition: none;
  }
}
```
