## 1. Distributed Cache Coherence

### 1.1 Problem Statement
In horizontally scaled deployments, multiple application nodes may each hold a local copy of cached data. Without a coherence protocol, stale reads and cache drift degrade correctness—especially under write-heavy workloads.

### 1.2 Coherence Protocols

#### A. Invalidation-Based Coherence (Preferred for Read-Heavy Workloads)
| Component | Mechanism |
|-----------|-----------|
| **Cache Invalidation Bus** | Pub/sub channel (Redis Pub/Sub, Kafka, NATS) carrying `{key, version, origin_node}` tuples |
| **Version Vector** | Each cache entry carries a Lamport-clock or hybrid-logical-clock version; nodes reject entries with `version < local_version` |
| **TTL with Active Invalidation** | Default TTL of 5–15 minutes; invalidation events zero-out TTL eagerly |
| **Originator Exclusion** | The node that performed the write skips invalidation for its own event (it already has the fresh value) |

```
┌──────────┐     Write     ┌──────────┐
│  Node A   │──────────────▶│  DB (SoT) │
│ (writer)  │               │           │
└─────┬─────┘               └───────────┘
      │ Invalidation Event
      │ {key: "user:42", version: 27, origin: "A"}
      ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Node B      │    │  Node C      │    │  Node D      │
│ Evicts key   │    │ Evicts key   │    │ Evicts key   │
│ on next read │    │ on next read │    │ on next read │
└─────────────┘    └─────────────┘    └─────────────┘
```

#### B. Update-Based Coherence (Preferred for Write-Heavy, Low-Latency Reads)
Instead of invalidations, the bus carries the **full value** or a **delta patch**. Nodes apply the update into their local cache directly. This trades bus bandwidth for zero cache-miss penalty on the next read.

#### C. Lease-Based Coherence
Nodes acquire a **lease** (time-bound exclusive or shared right) on cache entries. The lease manager (often Redis or etcd) enforces:
- **Read lease:** Shared; multiple nodes can hold simultaneously.
- **Write lease:** Exclusive; invalidates all read leases before granting.

### 1.3 Implementation Blueprint (Redis-Backed Invalidation Bus)

```python
# cache_coherence.py
import redis
import threading
import hashlib
import time

class CoherentLocalCache:
    """L1 cache with distributed coherence via Redis pub/sub."""

    def __init__(self, redis_client: redis.Redis, node_id: str,
                 max_size: int = 10_000, default_ttl: int = 300):
        self.redis = redis_client
        self.node_id = node_id
        self.default_ttl = default_ttl
        self._store: dict = {}          # key → (value, version, expires_at)
        self._lock = threading.RLock()
        self._subscriber = redis_client.pubsub()
        self._subscriber.subscribe("cache:invalidation")
        self._thread = threading.Thread(target=self._listen, daemon=True)
        self._thread.start()

    def get(self, key: str):
        with self._lock:
            entry = self._store.get(key)
            if entry is None:
                return None  # Cache miss → fetch from L2/DB
            value, version, expires_at = entry
            if time.monotonic() > expires_at:
                del self._store[key]
                return None
            return value

    def set(self, key: str, value, version: int = None, origin_node: str = None):
        version = version or int(time.time() * 1000)
        origin = origin_node or self.node_id
        with self._lock:
            self._store[key] = (value, version, time.monotonic() + self.default_ttl)
        # Publish invalidation event so peers know a new version exists
        self.redis.publish("cache:invalidation",
            f"{key}|{version}|{origin}".encode())

    def _listen(self):
        for message in self._subscriber.listen():
            if message["type"] != "message":
                continue
            key, version_str, origin = message["data"].decode().split("|")
            version = int(version_str)
            if origin == self.node_id:
                continue  # We wrote it; skip our own event
            with self._lock:
                entry = self._store.get(key)
                if entry and entry[1] < version:
                    del self._store[key]  # Evict; next read will fetch fresh
```

### 1.4 Trade-off Matrix

| Strategy | Consistency | Latency | Complexity | Bandwidth |
|----------|-------------|---------|------------|-----------|
| Invalidation | Eventual (seconds) | Low (reads hit local) | Medium | Low |
| Update-push | Strong (near real-time) | Lowest | High | High |
| Lease-based | Strong (bounded staleness) | Low | High | Low |
| TTL-only | Weak | Lowest | Lowest | None |

**Recommendation:** Start with invalidation-based coherence + version vectors. Graduate to update-push only for hot keys where read latency must be zero and write throughput justifies the bandwidth cost.

## 2. Write-Through with Change Data Capture (CDC)

### 2.1 Architecture Overview

Write-through caching ensures that every write to the database is synchronously reflected in the cache. Paired with CDC, the cache stays consistent even when writes originate from sources other than the application layer (batch jobs, other services, direct DB mutations).

```
┌──────────┐   Write    ┌───────────┐   CDC Stream    ┌──────────────┐
│  App     │───────────▶│  Primary   │───────┬────────▶│  Debezium /  │
│  Server  │◀── cache ──│  Database  │       │         │  Kafka Conn. │
└────┬─────┘   update   └───────────┘       │         └──────┬───────┘
     │                                      │                │
     │ Write-through                        │                │ Change Events
     ▼                                      ▼                ▼
┌──────────┐                        ┌─────────────────────────────┐
│  Redis   │◀─────── CDC Worker ────│  Kafka / Pulsar Topic       │
│  (L2)    │     updates cache      │  (cdc.{schema}.{table})     │
└──────────┘                        └─────────────────────────────┘
```

### 2.2 Write-Through Flow

```
1. Application receives write request (e.g., PATCH /users/42)
2. BEGIN transaction
3. UPDATE users SET email = 'new@example.com' WHERE id = 42
4. SET cache:user:42 → serialized_user_record (Redis SET with same transaction scope,
   or idempotent retry queue on failure)
5. COMMIT transaction
6. Return 200 OK to client
```

**Failure Modes & Mitigations:**

| Failure | Mitigation |
|---------|------------|
| Cache write fails after DB commit | Outbox pattern: write cache-update event to outbox table in same DB transaction; async worker retries |
| DB commit succeeds, cache write lost | CDC worker catches the DB change and repairs cache within seconds |
| Cache write succeeds, DB commit fails | Roll back transaction; delete the just-written cache key (or rely on TTL) |
| Network partition between app and Redis | Circuit-breaker on Redis; fall back to DB read; alert on open circuit |

```python
import json
from kafka import KafkaConsumer

CACHE_KEY_MAP = {
    ("public", "users"):    lambda row: f"user:{row['id']}",
    ("public", "orders"):   lambda row: f"order:{row['id']}",
    ("public", "products"): lambda row: f"product:{row['id']}",
}

consumer = KafkaConsumer(
    "cdc.public.users", "cdc.public.orders", "cdc.public.products",
    bootstrap_servers="kafka:9092",
    group_id="cdc-cache-worker",
    value_deserializer=lambda m: json.loads(m.decode("utf-8")),
)

for msg in consumer:
    event = msg.value
    payload = event.get("payload", {})
    op = payload.get("op")  # c, u, d, r
    source = payload.get("source", {})
    schema_table = (source.get("schema"), source.get("table"))
    key_fn = CACHE_KEY_MAP.get(schema_table)

    if key_fn is None:
        continue

    if op == "d":  # Delete
        after = payload.get("before")
        cache_key = key_fn(after) if after else None
        if cache_key:
            redis_client.delete(cache_key)
    elif op in ("c", "u", "r"):  # Create, Update, Snapshot/Read
        after = payload.get("after")
        if after:
            cache_key = key_fn(after)
            redis_client.setex(cache_key, TTL_SECONDS, json.dumps(after))
```

## 3. Cache Warming Strategies

### 3.1 Strategy Taxonomy

```
CACHE WARMING
├── Startup Warming
│   ├── Snapshot-based: Load precomputed cache dump
│   ├── Replay-based: Replay recent read-logs against fresh cache
│   └── Hot-key-only: Warm top-N% keys by access frequency
│
├── Proactive Warming
│   ├── Predictive: ML model forecasts future hot keys
│   ├── Scheduled: Cron-driven warm of known hotspots (e.g., 6 AM for morning traffic)
│   └── Event-driven: Deployment, failover, scale-out triggers warm cycle
│
└── Reactive Warming
    ├── Cache-miss amplification: On miss, fetch + preload related entities
    ├── Thundering-herd prevention: Single-flight / request coalescing
    └── TTL-extension: Refresh hot keys before expiry (early recompute)
```

### 3.2 Startup Warming: Snapshot-Based

```python
# cache_warmer.py
import redis
import json
from datetime import datetime, timedelta

class SnapshotCacheWarmer:
    """Loads a pre-built cache snapshot on cold start."""

    def __init__(self, redis_client: redis.Redis, snapshot_path: str):
        self.redis = redis_client
        self.snapshot_path = snapshot_path

    def warm_from_snapshot(self, batch_size: int = 1000):
        """Stream snapshot into Redis with pipelining."""
        with open(self.snapshot_path, "r") as f:
            pipe = self.redis.pipeline(transaction=False)
            count = 0
            for line in f:
                record = json.loads(line)
                key = record["key"]
                value = record["value"]
                ttl = record.get("ttl", 3600)
                pipe.setex(key, ttl, json.dumps(value))
                count += 1
                if count % batch_size == 0:
                    pipe.execute()
                    pipe = self.redis.pipeline(transaction=False)
            pipe.execute()
        return count

# done
```

### 3.4 Single-Flight (Request Coalescing)

Prevents thundering herds where 10,000 concurrent requests all miss the same expired key:

```python
import threading
import functools

class SingleFlight:
    """Ensures only one caller fetches a key; others wait for the result."""

    def __init__(self):
        self._pending: dict[str, threading.Event] = {}
        self._results: dict[str, any] = {}
        self._lock = threading.Lock()

    def do(self, key: str, fn, *args, **kwargs):
        with self._lock:
            if key in self._pending:
                event = self._pending[key]
            else:
                self._pending[key] = event = threading.Event()

        if event is not self._pending.get(key):
            # We're a waiter, not the leader
            event.wait()
            return self._results[key]

        try:
            result = fn(*args, **kwargs)
            self._results[key] = result
            return result
        finally:
            with self._lock:
                event.set()
                del self._pending[key]
                # Keep result briefly for late waiters
                threading.Timer(5.0, lambda: self._results.pop(key, None)).start()
```

```python
# l1_cache.py
from theine import Cache
from datetime import timedelta

class L1Cache:
    """Process-local, high-performance L1 cache."""

    def __init__(self, max_size: int = 100_000):
        self._cache = Cache("tlfu", max_size)  # TinyLFU policy

    def get(self, key: str):
        return self._cache.get(key)  # Returns None on miss

    def set(self, key: str, value, ttl: timedelta = timedelta(minutes=5)):
        self._cache.set(key, value, ttl)

    def delete(self, key: str):
        self._cache.delete(key)
```

### 4.3 Two-Tier Read Path

```python
# two_tier_cache.py
import asyncio
import json
from typing import Optional

class TwoTierCache:
    """L1 (Theine, local) + L2 (Redis, cluster)."""

    def __init__(self, l1: L1Cache, redis_client):
        self.l1 = l1
        self.l2 = redis_client
        self.singleflight = SingleFlight()

    async def get(self, key: str, db_fetcher=None) -> Optional[any]:
        # 1. Check L1
        value = self.l1.get(key)
        if value is not None:
            return value

        # 2. Check L2 (with single-flight for hot-miss coalescing)
        async def fetch_l2():
            raw = await self.l2.get(key)
            if raw is None:
                return None
            return json.loads(raw)

        value = await self.singleflight.do_async(key, fetch_l2)
        if value is not None:
            self.l1.set(key, value)  # Promote to L1
            return value

        # 3. Fetch from DB if fetcher provided
        if db_fetcher:
            value = await db_fetcher(key)
            if value is not None:
                # Populate both tiers
                await self.l2.setex(key, 3600, json.dumps(value))
                self.l1.set(key, value)
            return value

        return None

    async def write_through(self, key: str, value, ttl: int = 3600):
        """Write-through: update L2, then L1 (coherence handled separately)."""
        await self.l2.setex(key, ttl, json.dumps(value))
        self.l1.set(key, value, timedelta(seconds=ttl))
```

| Tier | Policy | Rationale |
|------|--------|-----------|
| L1 | W-TinyLFU | Balances recency (recent items) and frequency (popular items); high hit rate with bounded memory |
| L2 (Redis) | allkeys-lru or allkeys-lfu | `allkeys-lru` for general workloads; `allkeys-lfu` when access patterns are highly skewed |
| L2 Alternative | volatile-ttl | When you need precise TTL-driven expiry and don't want to evict keys with infinite TTL |

```python
# metrics.py
from dataclasses import dataclass, field
from collections import defaultdict

class TierMetrics:
```

```yaml
          (sum(rate(cache_hit_total[1h])) + 1)
```

## Appendix A: Key Terminology

| Term | Definition |
|------|------------|
| **SoT (Source of Truth)** | The authoritative data store (primary database) |
| **CDC** | Change Data Capture — streaming DB changes to downstream systems |
| **W-TinyLFU** | Window TinyLFU — a modern caching eviction policy combining LRU and LFU |
| **Single-flight** | Deduplication pattern ensuring only one request fetches a given key |
| **Coherence bus** | Pub/sub channel carrying invalidation/update events between cache nodes |
| **Read-your-writes** | Consistency guarantee that a client always sees its own writes |
