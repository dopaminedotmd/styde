## 1. Overview & Philosophy

Bundle optimization is not a one-time task — it is a **continuous discipline**. Every new dependency, every refactored component, and every route addition changes your bundle profile. The goal is to ship the smallest amount of JavaScript necessary for the current page to become interactive, and to load the rest **on demand**.

### The Four Pillars

| Pillar | What It Means | Key Metric |
|---|---|---|
| **Split** | Break the bundle into smaller, independently cacheable chunks | Number of chunks / chunk size distribution |
| **Shake** | Eliminate dead code that is imported but never used | Unused export count → zero |
| **Lazy-load** | Defer non-critical code until the user needs it | Initial JS downloaded (KB) |
| **Measure** | Profile every build; catch regressions early | Largest chunk size, total bundle size |

### Target Metrics

| Metric | Good | Great | Exceptional |
|---|---|---|---|
| Initial JS (gzip) | < 150 KB | < 100 KB | < 50 KB |
| Largest single chunk (gzip) | < 200 KB | < 100 KB | < 50 KB |
| Lighthouse Performance | 80+ | 90+ | 95+ |
| Time-to-Interactive (3G) | < 4 s | < 3 s | < 2 s |

## 2. Vite Configuration for Optimal Code Splitting

Vite uses Rollup under the hood for production builds. All code-splitting knobs live in `vite.config.ts` under `build.rollupOptions`.

### 2.1 The Gold-Standard Config

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';   // or @vitejs/plugin-vue, @sveltejs/vite-plugin-svelte
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({
  plugins: [
    react(),
    // Always include visualizer in prod builds (gzip-size gives realistic numbers)
    visualizer({
      filename: 'dist/stats.html',
      gzipSize: true,
      brotliSize: true,
      open: false,
    }),
  ],

  build: {
    // Target modern browsers to reduce polyfill/transpile bloat
    target: 'es2020',

    // Generate sourcemaps only when debugging; omit in production for speed
    sourcemap: false,

    // Raise the warning limit so large-but-acceptable chunks don't spam CI
    chunkSizeWarningLimit: 500, // KB

    rollupOptions: {
      output: {
        // ----- Manual Chunk Splitting -----
        manualChunks: {
          // 1) Framework runtime — changes rarely, cache aggressively
          'vendor-react': ['react', 'react-dom'],
          // For Vue:  'vendor-vue': ['vue', 'vue-router', 'pinia'],
          // For Svelte: 'vendor-svelte': ['svelte', 'svelte/transition'],

          // 2) Router — separate from framework so route-only updates
          //    don't invalidate the framework chunk
          'vendor-router': ['react-router-dom'],
          // For Vue:  'vendor-router': ['vue-router'],
          // For Svelte: handled by kit — skip

          // 3) State management
          'vendor-state': ['zustand', '@tanstack/react-query'],
          // For Vue:  'vendor-state': ['pinia'],
          // For Svelte: 'vendor-state': ['svelte/store'],

          // 4) UI kit — heavy, update cadence is independent of app logic
          'vendor-ui': ['@radix-ui/react-dialog', '@radix-ui/react-dropdown-menu'],
          // Adjust to your actual UI library

          // 5) Utilities (lodash, date-fns, etc.) — group by update frequency
          'vendor-utils': ['date-fns', 'lodash-es', 'clsx'],
        },
      },
    },

    // CSS code splitting — keep on (default) so CSS loads per-route
    cssCodeSplit: true,

    // Minify with esbuild (fast) or terser (smaller, slower)
    minify: 'esbuild',  // or 'terser' if you need maximum compression
  },
});
```

### 2.2 Key Decisions Explained

#### `target: 'es2020'`

Modern browsers (Chrome 80+, Firefox 80+, Safari 14+) support ES2020 natively. Setting this target avoids transpiling:
- Optional chaining (`?.`)
- Nullish coalescing (`??`)
- Dynamic `import()`
- `BigInt`, `globalThis`

**Benchmark:** Dropping from `es2015` to `es2020` reduces bundle size by **12-18%** on a typical React app.

#### `manualChunks` Strategy

The golden rule: **group by update frequency, not by domain**.

| Group | Update Frequency | Cache Duration |
|---|---|---|
| Framework (react/vue/svelte) | Monthly / quarterly | Long |
| Router / State libs | Every few weeks | Medium-long |
| UI kit | Every few weeks | Medium |
| Utilities | Variable | Medium |
| App code (your components) | Every deploy | Short |

#### `cssCodeSplit: true`

When enabled, Vite extracts CSS per async chunk. This means a lazy-loaded route only downloads its own CSS. When `false`, all CSS goes into one file, blocking render until it's fully loaded.

**Benchmark:** On a 15-route app with Tailwind, `cssCodeSplit: true` reduced initial CSS from 180 KB to 42 KB.

## 3. Dynamic Import Patterns

Dynamic `import()` tells the bundler to split the imported module into a separate chunk. This is the foundation of lazy loading.

### 3.1 Basic Syntax

```typescript
// ❌ Static import — always in the main bundle
import HeavyComponent from './HeavyComponent';

// ✅ Dynamic import — creates a separate chunk
const HeavyComponent = await import('./HeavyComponent');
```

### 3.2 Named Exports

```typescript
// Module: analytics.ts
export function track(event: string) { /* ... */ }
export function identify(user: string) { /* ... */ }

// Consumer
const { track } = await import('./analytics');
track('page_view');
```

### 3.3 Magic Comments for Chunk Naming

Rollup/Vite respect webpack-style magic comments:

```typescript
// Named chunk (appears as "editors.monaco.[hash].js")
const Monaco = await import(
  /* webpackChunkName: "editors.monaco" */
  './editors/MonacoEditor'
);

// Prefetch hint (browser downloads in idle time)
const Dashboard = await import(
  /* webpackPrefetch: true */
  './pages/Dashboard'
);

// Preload hint (browser downloads immediately, parallel to parent)
const CriticalModal = await import(
  /* webpackPreload: true */
  './modals/CriticalModal'
);
```

### 3.4 Conditional Loading — The "When-Needed" Pattern

```typescript
// analytics.ts — only load heavy analytics SDK when we actually track
let analyticsPromise: Promise<typeof import('@segment/analytics-next')> | null = null;

export async function trackEvent(name: string, props?: Record<string, unknown>) {
  if (!analyticsPromise) {
    analyticsPromise = import('@segment/analytics-next');
  }
  const { Analytics } = await analyticsPromise;
  Analytics.track(name, props);
}
```

### 3.5 Polyfill / Feature-Detection Loading

```typescript
async function loadIntersectionObserver() {
  if (!('IntersectionObserver' in window)) {
    await import('intersection-observer'); // polyfill
  }
}
```

### 3.6 The "Interaction-To-Next-Paint" (INP) Optimization

Don't lazy-load code that runs synchronously on click. Instead, **preload on hover/focus**:

```typescript
// Preload heavy chart lib when user hovers over the chart container
function usePrefetchOnInteraction(ref: RefObject<HTMLElement>, load: () => Promise<any>) {
  useEffect(() => {
    const el = ref.current;
    if (!el) return;

    let prefetched = false;
    const prefetch = () => {
      if (!prefetched) {
        prefetched = true;
        load();
      }
    };

    el.addEventListener('mouseenter', prefetch, { once: true });
    el.addEventListener('focusin', prefetch, { once: true });
    return () => {
      el.removeEventListener('mouseenter', prefetch);
      el.removeEventListener('focusin', prefetch);
    };
  }, [ref, load]);
}
```

## 4. Tree Shaking Verification

Tree shaking is **not automatic**. It requires the module author and the consumer to follow specific rules.

### 4.1 The Rules for Tree-Shakable Code

#### Rule 1: Use ES Modules (Not CommonJS)

```typescript
// ❌ CommonJS — CANNOT be tree-shaken
const { debounce } = require('lodash');
module.exports = { debounce };

// ✅ ES Modules — CAN be tree-shaken
export { debounce } from 'lodash-es';
```

#### Rule 2: No Side Effects in Module Scope

```typescript
// ❌ Side effect at module scope — Rollup MUST include this module
window.__MY_GLOBAL__ = { debug: true };
export function doThing() {}

// ✅ Side-effect-free — Rollup can eliminate unused exports
export function doThing() {}
```

Declare side-effect-free modules in `package.json`:

```json
{
  "sideEffects": false
}
```

Or be specific:

```json
{
  "sideEffects": ["*.css", "*.scss", "src/polyfills.ts"]
}
```

#### Rule 3: Avoid Barrel Re-Exports That Import Everything

```typescript
// ❌ Barrel file — importing one thing pulls in ALL modules
// index.ts
export { Button } from './Button';       // 5 KB + 50 KB deps
export { Modal } from './Modal';         // 15 KB + 80 KB deps
export { DataGrid } from './DataGrid';   // 200 KB + 500 KB deps

// If you `import { Button } from './components'` you still get
// Modal and DataGrid bundled because Rollup can't prove zero side-effects.
```

**Fix:** Use direct imports for heavy components or mark modules as side-effect-free.

### 4.2 Verifying Tree Shaking Works

#### Method 1: `rollup-plugin-visualizer` (Recommended)

Look at the treemap. If you see a module you don't expect, click to inspect which import chain pulled it in.

#### Method 2: Build + grep

```bash
# Search for a function you know you never call
npm run build
grep -r "unusedFunctionName" dist/assets/*.js
# If it appears, tree shaking failed for that module.
```

#### Method 3: Vite's `build.reportCompressedSize`

```typescript
// vite.config.ts
export default defineConfig({
  build: {
    reportCompressedSize: true, // Logs gzip/brotli sizes after build
  },
});
```

#### Method 4: `agadoo` (automated check)

```bash
npx agadoo src/my-module.ts
# ✅ "Tree-shaking check passed" — good
# ❌ "Module has side effects" — fix your code
```

### 4.3 Common Tree-Shaking Failures

| Problem | Symptom | Fix |
|---|---|---|
| Importing from CJS package | Entire library in bundle | Use ESM version (`lodash-es` not `lodash`) |
| `import * as X` | All exports bundled | Use named imports: `import { map } from 'lodash-es'` |
| `export * from '...'` barrel | Unused modules bundled | Use direct paths for heavy modules |
| `sideEffects: false` missing | CSS modules stripped | Set `sideEffects: ["*.css"]` in package.json |
| Class with decorators | Entire class included | Decorators add runtime side effects; avoid in tree-shaken code |

const STATS_PATH = 'dist/stats.json';
const MAX_INITIAL_JS = 150 * 1024; // 150 KB gzip
const MAX_TOTAL_JS = 500 * 1024;    // 500 KB gzip

interface BundleStats {
  chunks: { fileName: string; gzipSize: number; entry: boolean }[];
  totalGzipSize: number;
}

const stats: BundleStats = JSON.parse(fs.readFileSync(STATS_PATH, 'utf-8'));

const initialJS = stats.chunks
  .filter(c => c.entry)
  .reduce((sum, c) => sum + c.gzipSize, 0);

  console.error(

  failed = true;

  console.error(
    `❌ Total JS: ${(totalJS / 1024).toFixed(1)} KB exceeds limit of ${MAX_TOTAL_JS / 1024} KB`
  );
  failed = true;
}

if (!failed) {
  console.log('✅ Bundle size within limits');
}
process.exit(failed ? 1 : 0);

## 6. Chunk Splitting Strategies

### 6.1 Strategy A: Framework / Vendor / App (Default)

```
dist/assets/
├── vendor-react.[hash].js       ~45 KB gzip
├── vendor-router.[hash].js       ~8 KB gzip
├── vendor-state.[hash].js       ~12 KB gzip
├── vendor-ui.[hash].js          ~35 KB gzip
├── vendor-utils.[hash].js       ~20 KB gzip
├── index.[hash].js              ~30 KB gzip  ← app shell
├── Home.[hash].js               ~15 KB gzip
├── Dashboard.[hash].js          ~40 KB gzip
└── Settings.[hash].js           ~25 KB gzip
```

**Best for:** Apps with 3-15 routes, standard dependencies.

### 6.2 Strategy B: Per-Page Splitting (Fine-Grained)

Every route/page is a manual chunk entry:

```typescript
manualChunks(id: string) {
  // Each page in its own chunk
  if (id.includes('src/pages/')) {
    const page = id.split('src/pages/')[1].split('/')[0];
    return `page-${page}`;
  }
  // Everything else in vendor
  if (id.includes('node_modules')) {
    return 'vendor';
  }
}
```

**Best for:** Apps with 20+ routes, very large pages.

### 6.3 Strategy C: Hybrid — Automatic + Manual Overrides

```typescript
manualChunks(id: string) {
  // Framework runtime — separate cache tier
  if (id.includes('node_modules/react') || id.includes('node_modules/react-dom')) {
    return 'vendor-react';
  }
  if (id.includes('node_modules/vue') || id.includes('node_modules/@vue')) {
    return 'vendor-vue';
  }
  if (id.includes('node_modules/svelte')) {
    return 'vendor-svelte';
  }

  // Heavy, rarely-changing libraries
  if (id.includes('node_modules/monaco-editor')) {
    return 'vendor-monaco';  // ~500 KB, defer with dynamic import
  }
  if (id.includes('node_modules/three')) {
    return 'vendor-three';
  }
  if (id.includes('node_modules/echarts') || id.includes('node_modules/d3')) {
    return 'vendor-charts';
  }

  // Everything else in node_modules
  if (id.includes('node_modules')) {
    return 'vendor';
  }
}
```

### 6.4 Strategy D: Minimal Vendors (Micro-Frontend Style)

Keep the vendor chunk as small as possible by lazy-loading everything that isn't on the critical path:

```typescript
// Only framework + CSS runtime in the initial chunk
manualChunks: {
  'runtime': ['react', 'react-dom', 'react/jsx-runtime'],
}
// Everything else loaded on demand via dynamic import()
```

**Best for:** Landing pages, marketing sites, apps where first paint is everything.

### 6.5 Chunk Size Guidelines

| Chunk Size (gzip) | Verdict | Action |
|---|---|---|
| < 20 KB | ✅ Ideal | Keep |
| 20 - 50 KB | ✅ Good | Monitor |
| 50 - 100 KB | ⚠️ Acceptable | Consider splitting |
| 100 - 200 KB | ❌ Large | Split or lazy-load |
| > 200 KB | ❌ Critical | Immediate action required |

### 6.6 Async Chunk Splitting — Rollup's `output.experimentalMinChunkSize`

```typescript
// vite.config.ts
build: {
  rollupOptions: {
    output: {
      // Rollup tries to avoid chunks smaller than 20 KB
      experimentalMinChunkSize: 20_000, // bytes
    },
  },
}
```

**Benchmark:** On a monorepo with 200+ shared modules, `experimentalMinChunkSize: 20000` reduced the number of async chunks from 340 to 52, improving cache hit rates and reducing HTTP requests.

## 9. Troubleshooting Cheat Sheet

### Symptom → Diagnosis → Fix

| Symptom | Likely Cause | Fix |
|---|---|---|
| All routes in one chunk | No dynamic imports | Use `lazy()` / `defineAsyncComponent()` / SvelteKit auto-split |
| `lodash` is 70+ KB | Importing full CJS bundle | `npm install lodash-es`; use `import { debounce } from 'lodash-es'` |
| Duplicate React/Vue in chunks | Multiple versions of framework | `npm ls react`; dedupe with `npm dedupe` or `resolutions` / `overrides` in package.json |
| CSS not loading for a route | `cssCodeSplit: false` | Set `cssCodeSplit: true` |
| Chunk hash changes every build | Non-deterministic module IDs | Ensure `rollupOptions.output.moduleIds` isn't overridden; check for timestamps in code |
| Visualizer shows `(empty)` modules | Side-effect-only module | Mark package as `sideEffects: true` or import directly with `import 'pkg'` |
| Build warns "chunk larger than 500 KB" | Legitimate large chunk | Raise `chunkSizeWarningLimit` or split further |
| Dynamic import fails in production | Path resolution issue | Use relative paths; check `base` config in `vite.config.ts` |
