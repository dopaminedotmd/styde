### 1.2 Key Design Decisions

| Decision | Rationale |
|---|---|
| `scaled_float` for prices | Precise decimal arithmetic, no floating-point drift. Scaling factor 100 = cent-level precision |
| `nested` for variants/attributes | Preserves relationships; prevents cross-object matching in queries |
| `keyword` sub-fields for facets | Exact-match aggregations require `keyword` type |
| `search_analyzer` on autocomplete fields | Prevents edge n-gram tokenization of the query itself |
| `dynamic: strict` | Rejects unmapped fields; prevents mapping explosion |
| `index: false` on image URLs | URLs aren't searched; saves disk and memory |
| `doc_values: true` on `search_boost` | Enables script-based boosting in function_score queries |

## 2. Full-Text Search Queries with Boosting

### 2.1 Multi-Field Search with Tiered Boosting

This is the core e-commerce search query. Fields are weighted by business importance: matching the product name scores highest, followed by brand, category, and description.

```json
POST /products/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "multi_match": {
            "query": "wireless bluetooth headphones",
            "type": "best_fields",
            "fields": [
              "name^10",
              "name.synonym^9",
              "name.shingle^7",
              "name.exact^20",
              "brand^6",
              "brand.keyword^12",
              "category_path^4",
              "tags^3",
              "short_description^2",
              "description^1"
            ],
            "operator": "or",
            "fuzziness": "AUTO",
            "prefix_length": 2,
            "tie_breaker": 0.3
          }
        }
      ],
      "filter": [
        { "term": { "is_active": true } },
        { "term": { "availability": "in_stock" } }
      ]
    }
  },
  "min_score": 0.5,
  "size": 20,
  "from": 0
}
```

**Boosting Strategy Breakdown:**

| Field | Boost | Reasoning |
|---|---|---|
| `name.exact^20` | Highest | Exact name match = user intent certainty |
| `brand.keyword^12` | Very High | Exact brand queries (e.g., "Nike", "Apple") |
| `name^10` | High | Primary product title relevance |
| `name.synonym^9` | High | Synonym matches still strong intent |
| `name.shingle^7` | Medium-High | Phrase-aware matching ("wireless headphones") |
| `brand^6` | Medium | Brand mention in broader context |
| `category_path^4` | Medium | Category relevance signal |
| `tags^3` | Low-Medium | Supplementary metadata |
| `short_description^2` | Low | Secondary content |
| `description^1` | Lowest | Long-form content; least intent-strong |

### 2.2 Phrase-Prioritized Query (Precision Mode)

Use when exact phrase matching is critical (e.g., "iPhone 15 Pro Max 256GB"):

```json
POST /products/_search
{
  "query": {
    "bool": {
      "should": [
        {
          "match_phrase": {
            "name": {
              "query": "iPhone 15 Pro Max",
              "boost": 30,
              "slop": 2
            }
          }
        },
        {
          "match_phrase": {
            "name.shingle": {
              "query": "iPhone 15 Pro Max",
              "boost": 20
            }
          }
        },
        {
          "multi_match": {
            "query": "iPhone 15 Pro Max",
            "fields": ["name^8", "description^1"],
            "type": "best_fields",
            "fuzziness": "AUTO"
          }
        }
      ],
      "minimum_should_match": 1
    }
  }
}
```

### 2.3 Cross-Field Query (Broad Match Mode)

For ambiguous queries where term distribution across fields is unknown:

```json
POST /products/_search
{
  "query": {
    "multi_match": {
      "query": "red running shoes size 10",
      "type": "cross_fields",
      "fields": [
        "name^8",
        "description^2",
        "tags^3",
        "attributes.display_value^5"
      ],
      "operator": "and",
      "tie_breaker": 0.5
    }
  }
}
```

**`cross_fields`** treats the query as if all fields were one big field — ideal when query terms might appear in different fields (e.g., "red" in attributes, "shoes" in name).

### 2.4 Function Score Query: Business Metrics Integration

Combine text relevance with business signals (popularity, margin, recency):

```json
POST /products/_search
{
  "query": {
    "function_score": {
      "query": {
        "bool": {
          "must": [
            {
              "multi_match": {
                "query": "summer dress",
                "fields": ["name^10", "description^2", "tags^3"],
                "fuzziness": "AUTO"
              }
            }
          ],
          "filter": [
            { "term": { "is_active": true } }
          ]
        }
      },
      "functions": [
        {
          "field_value_factor": {
            "field": "sales_count",
            "factor": 0.1,
            "modifier": "log1p",
            "missing": 0
          },
          "weight": 2
        },
        {
          "field_value_factor": {
            "field": "rating",
            "factor": 0.5,
            "modifier": "none",
            "missing": 0
          },
          "weight": 1.5
        },
        {
          "gauss": {
            "created_at": {
              "origin": "now",
              "scale": "30d",
              "decay": 0.5
            }
          },
          "weight": 1
        },
        {
          "filter": { "term": { "promotion.is_on_sale": true } },
          "weight": 3
        }
      ],
      "score_mode": "sum",
      "boost_mode": "multiply",
      "max_boost": 42
    }
  }
}
```

**Tuning Parameters:**

- `score_mode: sum` — all function scores add together before being multiplied with the query score
- `boost_mode: multiply` — final score = query_score × function_score
- `max_boost: 42` — prevents runaway scores
- `log1p` modifier on sales — dampens extreme outliers (viral products)

## 4. Autocomplete with Edge N-Grams

### 4.1 Search-As-You-Type Architecture

The autocomplete pipeline has two phases:

1. **Index time:** Tokenize product names into edge n-grams (prefix fragments)
2. **Query time:** Match user keystrokes against the edge n-grams WITHOUT n-gramming the query

#### Index-Time Analyzer (already in mapping above)

```json
"autocomplete_index": {
  "type": "custom",
  "tokenizer": "edge_ngram_tokenizer",
  "char_filter": ["html_strip"],
  "filter": ["lowercase", "asciifolding"]
}
```

#### Query-Time Analyzer

```json
"autocomplete_search": {
  "type": "custom",
  "tokenizer": "standard",
  "char_filter": ["html_strip"],
  "filter": ["lowercase", "asciifolding"]
}
```

### 4.2 Autocomplete Query (Frontend Keystroke Handler)

```json
POST /products/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "match": {
            "name.autocomplete": {
              "query": "wire",
              "analyzer": "autocomplete_search",
              "operator": "and"
            }
          }
        }
      ],
      "filter": [
        { "term": { "is_active": true } },
        { "term": { "availability": "in_stock" } }
      ]
    }
  },
  "_source": ["id", "name", "brand.keyword", "price", "images.primary"],
  "size": 8,
  "sort": [
    { "sales_count": { "order": "desc" } },
    { "_score": { "order": "desc" } }
  ]
}
```

**Why this works:** "wire" typed by user matches edge n-gram tokens like `wi`, `wir`, `wire` generated from "wireless" at index time, but the query isn't n-grammed — the `search_analyzer` is `autocomplete_search` (standard tokenizer).

```json
      }
    }
  }
}
```

### 4.5 Performance Considerations for Edge N-Grams

| Concern | Mitigation |
|---|---|
| Index size bloat | Limit `max_gram` to 20; use only on name/brand fields |
| Slow indexing | Set `refresh_interval: 30s` during bulk indexing |
| High memory usage | Use separate "search" analyzer at query time to avoid n-gramming queries |
| Irrelevant matches | Combine with `minimum_should_match` and `prefix_length: 2` |
| Multi-word queries | Use `operator: "and"` to require all query tokens |

### 5.5 Tuning Checklist

| Step | Action |
|---|---|
| 1. Baseline | Record MRR/NDCG for top-50 queries before tuning |
| 2. Synonyms | Add missing synonym mappings; measure impact |
| 3. Boost weights | A/B test field boosts; small increments (e.g., 8 → 10) |
| 4. Fuzziness | Tune `prefix_length` (higher = stricter) and `fuzziness: AUTO` |
| 5. Stop words | Remove stop words? Test both; some queries need them ("The North Face") |
| 6. Stemming | Validate stemming doesn't over-stem (e.g., "running" → "run" is fine; "organization" → "organ" is not) |
| 7. Business signals | Add function_score with sales/rating/margin; monitor for relevance hijacking |
| 8. Click-through rate | Integrate CTR data as a ranking signal via function_score or learning-to-rank |
| 9. A/B test | Always validate changes against a holdout set of queries |
| 10. Monitor | Set up zero-results alerts; track p99 latency |

### 5.6 Common Relevance Problems & Fixes

| Problem | Symptom | Fix |
|---|---|---|
| **Stemming overaggression** | "running shoes" matches "run" but not "running" products | Use `keyword` sub-field for exact matches; reduce stemmer aggressiveness |
| **Fuzzy noise** | "bat" matches "cat", "hat", "bag" | Increase `prefix_length` to 2; reduce `fuzziness` to `1` |
| **Boosting imbalance** | Description matches outrank name matches | Audit field boosts; ensure name boost is 5-10x description |
| **Empty results** | Zero hits for valid queries | Check tokenization; add synonyms; lower `minimum_should_match` |
| **Popularity hijacking** | Irrelevant bestsellers dominate all queries | Cap function_score boost; use `boost_mode: multiply` with lower function weights |
| **Slow queries** | >200ms p99 latency | Profile with Profile API; reduce shard count; use `search_as_you_type` instead of edge n-grams |

### 6.6 Meilisearch Relevance Tuning

Meilisearch uses a built-in ranking rules pipeline (evaluated in order):

```
words → typo → proximity → attribute → sort → exactness
```

#### Custom Ranking Rules

```bash
curl -X PATCH 'http://localhost:7700/indexes/products/settings/ranking-rules' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer MASTER_KEY' \
  -d '[
    "words",
    "typo",
    "proximity",
    "attribute",
    "sort",
    "exactness",
    "sales_count:desc",
    "rating:desc",
    "review_count:desc"
  ]'
```

**Rule Explanation:**

| Rule | Behavior |
|---|---|
| `words` | Documents with all query terms rank first, then decreasing match count |
| `typo` | Fewer typos rank higher; zero typos first |
| `proximity` | Terms closer together in document rank higher |
| `attribute` | Terms in higher-priority attributes (name > brand > description) rank higher |
| `sort` | Applies user-specified sort criteria |
| `exactness` | Exact matches (same position, same attribute) rank above partials |
| `sales_count:desc` | Custom: higher-selling products surface first |
| `rating:desc` | Custom: higher-rated products break ties |

#### Attribute Ranking Order

```bash
curl -X PATCH 'http://localhost:7700/indexes/products/settings/searchable-attributes' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer MASTER_KEY' \
  -d '[
    "name",
    "brand",
    "tags",
    "category",
    "description",
    "sku"
  ]'
```

The order matters: a match in `name` outranks an identical match in `description`.

## 7. Comparison Matrix

| Feature | Elasticsearch | Meilisearch |
|---|---|---|
| **Language** | Java | Rust |
| **Configuration** | Extensive (mappings, analyzers, settings) | Minimal (auto-detected) |
| **Query DSL** | JSON-based, highly expressive | Simple REST, filter strings |
| **Relevance Tuning** | BM25, function_score, script_score, custom analyzers | Built-in ranking rules + custom sort |
| **Faceting** | Terms, range, histogram, nested, filters, significant_terms | Built-in `facets` parameter |
| **Autocomplete** | Edge n-gram analyzers, completion suggester | Built-in prefix search (no config needed) |
| **Typo Tolerance** | Fuzzy queries (manual config) | Built-in (automatic) |
| **Synonyms** | Custom analyzer with synonym file/graph | REST API or settings endpoint |
| **Geosearch** | geo_point + geo_shape queries | `_geoRadius` filter + `_geoPoint` sort |
| **Vector/Hybrid Search** | dense_vector + kNN (8.x+) | Hybrid search with embedders (v1.10+) |
| **Scaling** | Horizontal (shards, clusters, cross-cluster) | Vertical first; multi-node (cloud) |
| **Max Documents** | Billions | ~100M per index |
| **RAM Usage** | High (JVM heap: 4-32+ GB typical) | Low (~100-500 MB typical) |
| **Disk Usage** | Medium-high (inverted indices, doc values, stored fields) | Low (LMDB-based) |
| **DevOps Complexity** | High (cluster management, index lifecycle, snapshot policies) | Low (single binary, managed cloud option) |
| **Best For** | Large catalogs, complex relevance, custom pipelines | Small-medium catalogs, fast time-to-market, developer experience |
| **Pricing** | Free (self-hosted) / Elastic Cloud ($95+/mo) | Free (self-hosted) / Meilisearch Cloud ($30+/mo) |
| **Learning Curve** | Steep (2-4 weeks to production-ready) | Shallow (1-3 days to production-ready) |

### Decision Guide

**Choose Elasticsearch when:**
- Catalog exceeds 10M products
- Custom relevance modeling is critical (learning-to-rank, complex function scoring)
- You need advanced analytics (significant_terms, matrix aggregations)
- You have dedicated search engineering resources
- Multi-tenancy or strict security/RBAC is required
- You need join-like operations (nested, parent-child)

**Choose Meilisearch when:**
- Catalog is under 10M products
- Fast launch is the priority (hours, not weeks)
- Team lacks dedicated search expertise
- Predictable low-latency (<50ms) is essential
- Typo tolerance and prefix search must work out-of-box
- You want built-in faceting without aggregation DSL
- Operational simplicity matters (single binary, minimal tuning)

### Migration Path: Meilisearch → Elasticsearch

Startups often begin with Meilisearch and migrate to Elasticsearch as they scale. Recommended trigger points:

1. **Catalog >5M products**: Meilisearch performance degrades on single-node
2. **Need complex nested faceting**: e.g., "show me shoes available in size 10 AND color red"
3. **Custom relevance models**: When built-in ranking rules aren't enough
4. **Multi-language search**: Elasticsearch has richer language analyzers
5. **Real-time analytics on search data**: Elasticsearch's aggregation framework
