## 1. OAuth 2.1 with DPoP (Demonstration of Proof-of-Possession)

### 1.1 Architecture Overview

OAuth 2.1 consolidates the best practices from OAuth 2.0, removing insecure grants (implicit, ROPC) and mandating PKCE. DPoP (RFC 9449) adds sender-constraining to access tokens, preventing token replay attacks even when TLS is terminated at a reverse proxy.

```
┌──────────┐                    ┌─────────────────┐                    ┌──────────────┐
│  Client  │                    │  Auth Server     │                    │  Resource    │
│  (SPA/   │                    │  (StydeForge    │                    │  Server      │
│  Mobile) │                    │   Identity)      │                    │              │
└────┬─────┘                    └───────┬─────────┘                    └──────┬───────┘
     │                                  │                                     │
     │ 1. Generate DPoP key pair        │                                     │
     │    (ES256/RSA)                   │                                     │
     │                                  │                                     │
     │ 2. POST /token                   │                                     │
     │    DPoP: <proof-jwt>             │                                     │
     │    + code_verifier               │                                     │
     │    + authorization_code          │                                     │
     │─────────────────────────────────>│                                     │
     │                                  │                                     │
     │ 3. Validate DPoP proof:          │                                     │
     │    - Signature with jkt thumbprint│                                    │
     │    - htm (method) = POST         │                                     │
```

### 1.3 Nonce Management Strategy

```
┌───────────────────────────────────────────────────────────────┐
│                    Nonce Lifecycle                            │
├───────────────────────────────────────────────────────────────┤
│  1. Client requests /token (no nonce present)                 │
│     → Server generates nonce, returns in DPoP-Nonce header    │
│     → Returns use_dpop_nonce error (HTTP 401)                 │
│                                                               │
│  2. Client retries with nonce in DPoP proof                   │
│     → Server validates nonce hasn't been used                 │
│     → Marks nonce as consumed in Redis (TTL = 24h)            │
│     → Returns fresh nonce for next request                    │
│                                                               │
│  3. Resource Server nonce:                                    │
│     → RS maintains its own nonce pool                         │
│     → AS provides nonce in WWW-Authenticate header            │
│     → Nonce is scoped to (client_id, resource_server) tuple   │
└───────────────────────────────────────────────────────────────┘
```

### 1.4 Key Implementation Decisions

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| Key algorithm | ES256 (P-256) | Smaller signatures than RSA, WebCrypto native support |
| Key rotation | Per-session ephemeral | Limit window of compromise; regenerate for each auth session |
| ath claim | SHA-256 of access token | Binds DPoP proof to specific token, prevents proof cross-use |
| Nonce storage | Redis with TTL | Global, fast, auto-expiring; Redis Cluster for HA |
| Clock skew tolerance | ±60 seconds | Accommodates NTP drift; iat validation window |
| Proof replay cache | LRU in-memory + Redis fallback | `jti` deduplication for the nonce lifetime window |

### 1.5 OAuth 2.1 Specific Mandates

```
OAuth 2.1 vs OAuth 2.0 — Breaking Changes:

  REMOVED from OAuth 2.0:
    ✗ Implicit grant (response_type=token)     → Use Auth Code + PKCE
    ✗ ROPC grant (password)                    → Use device flow or CIBA
    ✗ Bearer tokens in query string            → Only in Authorization header
    ✗ Refresh token rotation optional          → Now MANDATORY
    ✗ Native app redirect URIs w/o proof key   → PKCE REQUIRED everywhere

  ADDED in OAuth 2.1:
    ✓ PKCE (S256) mandatory for all clients
    ✓ Sender-constraining via DPoP or mTLS
    ✓ Refresh token rotation with reuse detection
    ✓ Exact redirect_uri matching (no wildcards)
    ✓ Resource indicators (resource=) for multi-RS deployments
```

## 3. Step-Up Authentication

### 3.1 Concept

Step-up authentication requires the user to re-authenticate (or provide a stronger factor) when accessing sensitive resources, even if they already hold a valid session. This is critical for transactions like changing passwords, viewing sensitive data, or initiating financial operations.

```
┌─────────────────────────────────────────────────────────────────┐
│                     ACR (Authentication Context Class) Levels   │
├───────┬────────────────────────────────┬────────────────────────┤
│ Level │ Meaning                        │ Method                 │
├───────┼────────────────────────────────┼────────────────────────┤
│ acr0  │ No authentication              │ N/A (public endpoint)  │
│ acr1  │ Basic auth (password/MFA       │ Password + TOTP/       │
│       │ within last 24h)               │ SMS/Email OTP          │
│ acr2  │ Phishing-resistant MFA         │ FIDO2/WebAuthn         │
│       │ (hardware key, within 15 min)  │ hardware key           │
│ acr3  │ Hardware-bound + attestation   │ FIDO2 w/ attestation   │
│       │ verified (within 5 min)        │ + biometric check      │
└───────┴────────────────────────────────┴────────────────────────┘

Example: Transfer >$10,000 requires acr3 step-up
         View balance requires acr1
         View transaction history requires acr1
         Add new payee requires acr2
         Change password requires acr2
```

### 3.4 Resource Server Enforcement Middleware

```typescript
// Step-up enforcement middleware (conceptual)
interface StepUpConfig {
  path: string;           // e.g., "/admin/users"
  methods: string[];      // e.g., ["GET", "POST"]
  requiredAcr: string;    // e.g., "acr2"
  maxAgeSeconds?: number; // e.g., 300 (5 minutes)
}

const stepUpRules: StepUpConfig[] = [
  { path: "/admin/*",           methods: ["*"], requiredAcr: "acr2", maxAgeSeconds: 900 },
  { path: "/api/transfers",     methods: ["POST"], requiredAcr: "acr3", maxAgeSeconds: 300 },
  { path: "/api/account/email", methods: ["PUT"], requiredAcr: "acr2", maxAgeSeconds: 600 },
  { path: "/api/account/delete", methods: ["DELETE"], requiredAcr: "acr3", maxAgeSeconds: 120 },
];

function enforceStepUp(req, token, rule: StepUpConfig): boolean {
  // 1. Check ACR level
  const tokenAcr = parseAcrLevel(token.acr);
  const requiredAcr = parseAcrLevel(rule.requiredAcr);
  if (tokenAcr < requiredAcr) return false;

  // 2. Check authentication recency
  if (rule.maxAgeSeconds && token.auth_time) {
    const age = Date.now() / 1000 - token.auth_time;
    if (age > rule.maxAgeSeconds) return false;
  }

  return true;
}
```

## 5. Hardware Key Attestation

### 5.1 FIDO2/WebAuthn Attestation Overview

Attestation is the process by which a FIDO2 authenticator proves its provenance — cryptographically asserting its make, model, and firmware version to the relying party. This enables trust decisions based on the hardware authenticator's security properties.

```
┌──────────────────────────────────────────────────────────────┐
│                  Attestation Types (WebAuthn L2)             │
├──────────────┬───────────────────────────────────────────────┤
│ None         │ No attestation. Anonymized. Good for privacy. │
│ (none)       │ Used by most consumer WebAuthn.               │
├──────────────┼───────────────────────────────────────────────┤
│ Self         │ Authenticator signs with its own key. Basic   │
│ Attestation  │ integrity check. No manufacturer trust chain. │
│ (packed/     │                                                 │
│  self)       │                                                 │
├──────────────┼───────────────────────────────────────────────┤
│ Basic        │ Authenticator's unique key + certificate      │
│ Attestation  │ chain from manufacturer. Identifies the       │
│ (packed/     │ specific device (privacy concern).             │
│  basic)      │                                                 │
├──────────────┼───────────────────────────────────────────────┤
│ Attestation  │ Anonymized basic attestation. Certificate     │
│ CA (packed/  │ chain but no unique device identifier.        │
│  attca)      │ Uses batch keys.                               │
├──────────────┼───────────────────────────────────────────────┤
│ Enterprise   │ Full device identification with enterprise-   │
│ Attestation  │ managed authenticators. For corporate fleets. │
│ (packed/     │                                                 │
│  enterprise) │                                                 │
└──────────────┴───────────────────────────────────────────────┘
```

### 5.4 AAGUID Trust Store

The Authenticator Attestation GUID (AAGUID) identifies the authenticator model. Combine the FIDO MDS with a local allowlist/blocklist:

```yaml
attestation_policy:
  # Source of truth for authenticator metadata
  metadata_sources:
    - type: "fido_mds"
      url: "https://mds3.fidoalliance.org/"
      cache_ttl: 86400    # 24 hours
      api_token: "${FIDO_MDS_TOKEN}"
    - type: "local_override"
      path: "/etc/strydeforge/attestation-overrides.yaml"

  # Trust levels
  certification_levels:
    allowed:
      - "FIDO_CERTIFIED_L1"    # Minimum: FIDO L1
      - "FIDO_CERTIFIED_L1plus"
      - "FIDO_CERTIFIED_L2"    # Higher security
      - "FIDO_CERTIFIED_L3"    # Highest security
    disallowed:
      - "NOT_CERTIFIED"

  # Per-application attestation requirements
  policies:
    - name: "standard-users"
      require_attestation: false   # Accept any FIDO key
      allow_self_attestation: true
      attestation_type: ["none", "self", "basic", "attca"]

    - name: "admin-users"
      require_attestation: true
      attestation_type: ["attca", "basic", "enterprise"]
      allowed_aaguids:             # Explicit allowlist
        - "95442b2e-f15e-4def-b270-ef10f0e5a2a6"  # YubiKey 5 NFC
        - "ee882879-721c-4913-9775-3dfcce97072a"  # YubiKey 5C NFC
        - "73bb0cd4-e502-49b8-9c6f-b59445bf720b"  # YubiKey Bio
        - "6d44ba9b-f6ec-2e49-b930-0c8fe920cb73"  # YubiKey 5Ci

        - "9ddd1817-af5a-4672-a2b9-3e3ddfa05419"  # Apple App Attest key
      allowed_os_platforms:        # OS-level passkey providers
        - "apple"                   # iCloud Keychain (with Secure Enclave)
        - "google"                  # Google Password Manager
      minimum_certification: "FIDO_CERTIFIED_L2"

    - name: "high-security-operations"
```

### 5.6 Attestation Verification Server-Side (Rust/Go-like pseudocode)

```rust
// Conceptual attestation verification pipeline
struct AttestationResult {
    aaguid: Uuid,
    manufacturer: String,
    model: String,
    certification_level: CertificationLevel,
    attestation_type: AttestationType,
    trusted: bool,
    metadata: AuthenticatorMetadata,
    warnings: Vec<String>,
}

fn verify_attestation(
    attestation_obj: &AttestationObject,
    client_data_json: &[u8],
    expected_rp_id: &str,
    challenge: &[u8],
    policy: &AttestationPolicy,
    mds_cache: &MdsCache,
) -> Result<AttestationResult, AttestationError> {
    // 1. Verify client data
    let client_data = parse_client_data(client_data_json)?;
    verify_challenge(&client_data, challenge)?;
    verify_origin(&client_data, &policy.allowed_origins)?;

    // 2. Verify auth data
    let auth_data = &attestation_obj.auth_data;
    verify_rp_id_hash(auth_data, expected_rp_id)?;
    verify_user_present(auth_data)?;
    if policy.require_user_verification {
        verify_user_verified(auth_data)?;
    }

    // 3. Look up AAGUID in MDS
    let metadata = mds_cache.lookup(auth_data.attested_credential_data.aaguid)?;
    verify_certification_level(&metadata, &policy.minimum_certification)?;

    // 4. Verify attestation statement based on format
    match attestation_obj.fmt.as_str() {
        "packed" => {
            verify_packed_attestation(
                &attestation_obj.att_stmt,
                &auth_data,
                &client_data,
                &metadata.trust_anchors,
            )?
        }
        "apple" => {
            verify_apple_attestation(
              &attestation_obj.att_stmt,
              &auth_data,
              &client_data,
            )?
        }
        "android-safetynet" | "android-key" => {
            verify_android_attestation(
                &attestation_obj.att_stmt,
                &auth_data,
                &client_data,
            )?
        }
        "none" => {
            // No attestation — apply lower trust
            if !policy.allow_none_attestation {
                return Err(AttestationError::AttestationRequired);
            }
        }
        _ => return Err(AttestationError::UnsupportedFormat),
    }

    // 5. Check against allowlist/blocklist
    verify_aaguid_allowed(&metadata.aaguid, &policy)?;

    // 6. Verify certificate revocation status (OCSP stapling or CRL)
    verify_certificate_status(&attestation_obj.att_stmt.x5c)?;

    Ok(AttestationResult {
        aaguid: metadata.aaguid,
        manufacturer: metadata.manufacturer,
        model: metadata.model,
        certification_level: metadata.certification_level,
        attestation_type: attestation_obj.fmt.into(),
        trusted: true,
        metadata,
        warnings: vec![],
    })
}
```

## 7. Security Threat Model & Mitigations

| Threat | Attack Vector | Mitigation |
|--------|--------------|------------|
| Token replay | Attacker captures token, replays to RS | DPoP binding + ath claim + nonce validation |
| Token replay (TLS offload) | Attacker captures token after TLS termination | DPoP: proof JWT binds token to client's private key |
| Phishing | User tricked into entering credentials on fake site | Magic links (no passwords) + WebAuthn origin binding |
| Credential stuffing | List of breached passwords tried against system | Passwordless magic links eliminate passwords entirely |
| Device code phishing | Attacker generates device code, tricks user into approving | Show device name/IP in approval screen; user must match |
| Session hijacking | Attacker steals session token | Step-up re-authentication for sensitive ops; short token TTLs |
| Rogue authenticator | Attacker plugs in compromised FIDO key | Attestation verification against AAGUID allowlist + FIDO MDS |
| Magic link interception | Attacker reads victim's email | 10-minute expiry; single-use; IP change detection; page alerts |
| Refresh token theft | Persistent backend compromise | Rotation on use + reuse detection → full revocation |
| Man-in-the-middle | Attacker proxies connection | DPoP htu claim binds proof to exact URL; TLS cert pinning |
| Privacy (token tracking) | Tracking users via AAGUID | Use attestation type "none" or "attca" for privacy-preserving modes; batch-key attestation |
| Downgrade attack | Attacker forces lower acr | Server enforces per-endpoint acr; client cannot opt down |
