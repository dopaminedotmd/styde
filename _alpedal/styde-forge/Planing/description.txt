DASHBOARD_PHASE0/
================================================================================
Purpose: Complete specification for the Styde Dashboard v0 — a desktop-native
control panel built with Tauri (Rust backend) + React frontend. This is the
Phase 0 scope: a working prototype with shell, chat, model provider system,
agent monitoring, benchmark panel, and data layer.

--- 00_Overview (3 files) ---

Application_Architecture.md
  Full architectural blueprint of the Dashboard. Defines the three-layer stack:
  Tauri/Rust system layer, React/TypeScript UI layer, and IPC bridge. Covers
  process lifecycle, window management, event bus architecture, and how the
  Hermes CLI bridge connects the dashboard to the underlying agent system.
  Includes data flow diagrams for chat messages, agent spawns, and system
  events.

DASHBOARD_INDEX.md
  Master navigation index for ALL Phase 0 documents. A table of contents
  mapping every specification file to its section (01-10). Includes a
  dependency graph showing which documents must be read in which order
  to understand the full system.

Dashboard_Vision.md
  The "why" document. Defines the product vision: a unified desktop command
  center for managing multi-agent AI workflows. Target users are developers
  running Hermes/Forge locally. Covers design philosophy (speed over
  features, keyboard-first, dark-mode native), success criteria for Phase 0,
  and the north-star UX principles.

--- 01_Application_Shell (4 files) ---

Lifecycle_Management.md
  How the Tauri application starts, runs, and shuts down. Specifies startup
  sequence (splash screen -> preload checks -> main window -> system tray),
  lifecycle hooks (on_ready, on_focus, on_blur, on_close), and the state
  machine that governs app states (Loading, Ready, Minimized, Quitting).
  Includes error recovery procedures for crash scenarios.

Process_Control.md
  Defines how the Dashboard manages child processes (Hermes CLI, Forge agents,
  local model servers). Specifies process pools, health check intervals,
  graceful shutdown with SIGTERM, and automatic restart policies. Includes
  the ProcessManager API specification.

System_Tray_Integration.md
  System tray icon behaviour: right-click menu (Show/Hide, Status, Quit),
  tooltip updates reflecting agent status, and OS-specific considerations
  (Windows notification area vs macOS menu bar). Defines the tray icon state
  set (idle, busy, error, updating).

Window_Management.md
  Window geometry, multi-monitor support, window state persistence (position,
  size, maximized), and the "compact mode" that collapses the dashboard to a
  minimal sidebar. Includes protocols for programmatic window manipulation
  via IPC.

Design_System.md

Layout_Design.md
  Page layout specifications: the three-column layout (sidebar | main content
  | detail panel), responsive breakpoints, and the layout grid system.
  Includes wireframe descriptions for every major view: Dashboard Home,
  Agent Monitor, Chat, Settings, and Benchmark panels.

Onboarding_Flow.md
  First-run experience: splash screen, initial configuration wizard (API
  keys, model provider setup, workspace path), tooltip tour, and template
  selection. Designed to get a new user from zero to a running agent in
  under 2 minutes.

Purpose: Phase 1 of the Styde Forge system. Moves from prototype to
production-grade agent orchestration framework. Introduces the actual "Forge"
— a core loop that spawns, evaluates, and improves AI agents autonomously.

--- 00_Implementation_Overview (3 files) ---

IMPLEMENTATION_ORDER.md
  The build sequence for Phase 1: Forge Core first (bootstrap, circuit breaker,
  core loop), then Forge Spawn (blueprint loading, delegation, RAG), eval
  pipeline, improvement loop, dashboard shell, chat, and integration. Each
  step has estimated effort, dependencies, and acceptance criteria.

PHASE0_GAP_ANALYSIS.md
  A frank assessment of what Phase 0 didn't deliver. Lists missing features,
  incomplete specs, architectural debt, and decisions that need revisiting.
  Each gap is rated by severity (blocker/major/minor) and linked to a Phase 1
  deliverable that addresses it.

PHASE1_SCOPE.md
  The formal scope document: what's in (Forge Core, Forge Spawn, Eval Pipeline,
  Improvement Loop, Dashboard Shell, Chat, Integration) and what's explicitly
  out (multi-user, cloud, plugin ecosystem — those are Phase 2). Defines the
  "definition of done" for Phase 1.

--- 01_Forge_Core (4 files) ---

Bootstrap_Scripts.md
  How the Forge system initializes: environment checks (Python version, GPU
  availability, disk space), config loading, skill manifest loading, and
  dependency resolution. Includes the bootstrap sequence diagram and the
  script entry points (forge bootstrap, forge doctor).

Caveman_Ultra_Activation.md
  The "Caveman Ultra" mode — an ultra-compressed communication protocol for
  agent-to-agent messaging. Designed to minimize token usage (target ~75%
  reduction) by stripping all natural language fluff and using structured
  shorthand. Specifies the activation conditions, compression rules, and
  decompression layer.

Circuit_Breaker_Implementation.md
  Fault tolerance: circuit breaker pattern for API calls and agent operations.
  Three states: Closed (normal), Open (failing — fast-fail), Half-Open
  (testing recovery). Specifies thresholds (error count, time window), cooldown
  periods, and the implementation as a Python decorator/context manager.

Core_Loop_Implementation.md
  The heart of Forge: the main loop that orchestrates agent lifecycle.
  Pseudocode-level specification of the loop: 1) Load blueprint, 2) Spawn
  agent, 3) Monitor execution, 4) Evaluate output, 5) Improve blueprint,
  6) Loop or exit. Includes state management, error handling, and timeout
  logic.

--- 02_Forge_Spawn (3 files) ---

Blueprint_Loading_System.md
  How Forge loads agent blueprints from disk: YAML/JSON blueprint parsing,
  schema validation, dependency resolution (blueprint inheritance), and the
  blueprint cache. Defines the Blueprint schema (name, model, prompt, skills,
  tools, constraints, eval criteria).

Delegate_Task_Integration.md
  Integration with Hermes's delegate_task system: how Forge passes tasks to
  sub-agents, monitors completion, and collects results. Specifies the
  delegation API, timeout handling, and result aggregation.

RAG_Context_Injection.md
  How blueprints inject context from a knowledge base: chunking strategy,
  embedding model (text-embedding-3-small), vector store (Chroma/FAISS),
  retrieval parameters (top-k, similarity threshold), and context window
  management. Builds a persistent knowledge base from past agent runs.

--- 03_Forge_Eval (2 files) ---

Eval_Pipeline_Implementation.md
  The evaluation pipeline that runs after each agent execution: automated
  checks (schema validation, test pass/fail, performance metrics), LLM-as-judge
  scoring, and human review integration. Defines the EvalResult schema,
  scoring rubric, and pipeline stages.

Eval_Spec_Format.md
  The specification format for evaluation criteria: YAML-based eval specs
  with fields for name, description, eval_type (auto/llm/human), weight,
  pass_threshold, and parameters. Includes examples for common eval types
  (code quality, response accuracy, style compliance).

--- 04_Forge_Improve (3 files) ---

Blueprint_Improvement_Loop.md
  The improvement engine: takes eval results, identifies weaknesses, and
  generates blueprint modifications. Uses a feedback loop that compares
  expected vs actual outcomes, applies targeted improvements (prompt tuning,
  skill adjustments, constraint tightening), and re-runs the blueprint.

Skill_Extraction_System.md
  How Forge extracts reusable skills from successful agent runs: pattern
  detection, template generation, skill validation, and automatic registration
  in the skill catalog. The goal is to build an ever-growing library of
  proven agent capabilities.

Teacher_and_Persistence.md
  The "Teacher Agent" — a meta-agent that observes other agents, identifies
  learning opportunities, and stores insights in persistent memory. Covers
  the Teacher's knowledge base schema, observation model, and how lessons
  are applied to future blueprint executions.

--- 06_Dashboard_Shell (2 files) ---

foundation. Introduces Bayesian optimization, multi-judge evaluation, learning
systems, resource management, multi-agent orchestration, automation, and a

IMPLEMENTATION_ORDER.md
  Phase 2 build order: Bayesian Optimization first (foundation for intelligent
  tuning), then Multi-Judge Eval (better evaluation), Learning Systems
  (self-improvement), Resource Management (efficiency), Multi-Agent (scale),
  Automation (autonomy), Dashboard P1P2 (unified UI), and Integration (glue).

PHASE1_TO_PHASE2_TRANSITION.md
  What changes between Phase 1 and 2: architectural additions, new dependencies,
  breaking changes to Phase 1 APIs, migration strategies, and what gets
  deprecated. Ensures a smooth upgrade path from P1 to P2.

PHASE2_SCOPE.md
  Scope document: what's in (Bayesian optimization, multi-judge, learning
  systems, resource governor, multi-agent orchestration, automation engine,
  unified dashboard) and what's out (Phase 3 items like distributed execution,
  federated learning, marketplace).

--- 01_Bayesian_Optimization (1 file) ---

README.md
  Specification for the Bayesian optimization module. Uses Gaussian Process
  regression to optimize model parameters (temperature, top_p, frequency
  penalty, presence penalty) based on evaluation scores. Defines the
  acquisition function (Expected Improvement), parameter search space,
  and integration with the eval pipeline.

--- 02_Multi_Judge_Eval (1 file) ---

README.md
  Multi-judge evaluation system: runs the same output through multiple LLM
  judges (different models/prompts), aggregates scores using weighted voting
  or consensus algorithms, and produces a calibrated final score. Reduces
  individual judge bias and improves evaluation reliability.

--- 03_Learning_Systems (1 file) ---

README.md
  Meta-learning module: tracks patterns across all agent runs, identifies
  what works (success patterns) and what doesn't (failure modes), and
  automatically updates blueprints and skills. Implements few-shot learning
  from execution history and task clustering for transfer learning.

--- 04_Resource_Management (1 file) ---

README.md
  Resource governor: monitors and limits CPU, memory, GPU, and token usage
  across all running agents. Implements fair scheduling, priority queues,
  resource quotas per agent/workspace, and automatic scaling (spin down
  idle agents, throttle greedy ones). Includes cost tracking per model
  provider.

--- 05_Multi_Agent (1 file) ---

README.md
  Multi-agent orchestration: coordinates multiple agents working on the same
  or related tasks. Specifies communication protocols (agent-to-agent message
  passing), task decomposition (break task into sub-tasks for parallel agents),
  conflict resolution (when agents disagree), and consensus mechanisms.

--- 06_Automation (1 file) ---

README.md
  The automation engine: cron-job style scheduling for agent tasks, event-driven
  triggers (file change, webhook, time), pipeline composition (chain agents
  into workflows), and self-healing (retry failed tasks, alert on persistent
  failure). Users can set up recurring agent runs without manual intervention.

--- 07_Dashboard_P1P2 (1 file) ---

README.md
  The unified dashboard covering both Phase 1 (core monitoring) and Phase 2
  (advanced analytics). New views: optimization dashboard (Bayesian progress),
  evaluation leaderboard (multi-judge scores), learning dashboard (skill
  growth, pattern discovery), and resource dashboard (usage, costs, limits).
  All views share the Phase 1 shell and design system.

--- 08_Integration (1 file) ---

Week_Execution_Plan.md
  Phase 2 build schedule: week-by-week sprints for 12-14 weeks. Each week
  has specific deliverables from the scope areas, testing requirements, and
  integration checkpoints. Designed for a solo developer or small team.

V3.0_PHASE0/
================================================================================
Purpose: The most comprehensive phase — Phase 3 / v3.0 Phase 0. This is an
ambitious redesign of the entire Forge system with deep technical sophistication.
Adds Bayesian sampling (HMC, NUTS), formal state machines, hardware adaptation
layer, advanced eval pipelines, meta-learning, persistence safety, multi-agent

management, teacher agents, hooks/events systems, and RAG retrieval.

--- 02_Hardware (2 files) ---

Hardware_Adaptation_Layer.md
  Abstract hardware layer that adapts Forge to any hardware: detects GPU
  (NVIDIA CUDA, AMD ROCm, Apple Metal), CPU cores, RAM, and disk speed.
  Auto-configures model sizes, batch sizes, and parallelism based on
  available hardware. Supports headless/server environments without display.

Resource_Governor.md
  Advanced resource management: soft/hard limits per agent, dynamic
  allocation based on priority, GPU memory partitioning, token budgets
  (daily/weekly), cost-aware scheduling (cheapest model for easy tasks),

--- 03_Eval_Pipeline (6 files) ---

Automatic_Validation.md
  Automated validation rules for agent outputs: schema validation (JSON
  Schema), constraint satisfaction (logical rules), consistency checks
  (cross-reference with known facts), and output format verification.
  Runs before LLM-based evaluation to catch obvious issues early.

Bayesian_Weight_Optimization.md
  Bayesian optimization applied to evaluation weights: treats each evaluation
  criterion's weight as a hyperparameter, uses Gaussian Process regression
  to find the weight combination that best predicts human judgment. Closes
  the loop between automated and human evaluation.

Benchmark_Catalog.md
  Catalog of built-in benchmarks: coding benchmarks (HumanEval, MBPP),
  reasoning (GSM8K, MATH), language (MMLU, HellaSwag), and custom
  (Styde-specific task suites). Each benchmark entry defines the dataset,
  evaluation protocol, scoring system, and reference performance.

Bias_Calibration.md
  Bias detection and calibration in LLM judges: identifies systematic biases
  (verbosity preference, position bias, self-enhancement bias), measures
  bias magnitude, and applies calibration corrections. Includes a built-in
  bias audit tool that runs periodically.

Cross_Judge_Consensus.md
  Consensus algorithms for multi-judge evaluation: majority voting, weighted
  voting (by historical accuracy), Bayesian aggregation, and confidence-based
  filtering. Specifies when each algorithm is appropriate and how to handle
  judge disagreement.

LLM_as_Judge.md
